Capítulo 4

Del doble de prueba a la base de datos real

En esta fase se sustituye el repositorio falso por un adaptador MariaDB real. El objetivo didáctico es que el alumno vea una diferencia que en proyectos reales es constante: la lógica de negocio puede estar bien y, aun así, el acceso a datos puede estar mal.

Cómo se lanza la integración en Jenkins

Jenkinsfile
stage('Tests de integracion') {
    steps {
        script {
            docker.image('mariadb:11.4').withRun(...) { db ->
                docker.image('python:3.12-slim').inside("--link ${db.id}:db -u root:root") {
                    sh '''
                        export APP_DB_HOST=db
                        export APP_DB_PORT=3306
                        export APP_DB_NAME=gradebook
                        export APP_DB_USER=gradebook
                        export APP_DB_PASSWORD=gradebookpass
                        python scripts/wait_for_db.py
                        pytest tests/integration --junitxml=reports/integration.xml
                    '''
                }
            }
        }
    }
}

Hay dos contenedores efímeros: uno para la base y otro para el proceso Python de tests. El alumno ve así una topología mínima pero real de un pipeline de integración.

El bug que se introduce a propósito

En src/gradebook/mariadb_repository.py se cambia la consulta correcta:

Consulta correcta
WHERE student_name = %s

por esta variante errónea:

Consulta buggy
WHERE module_name = %s

Qué efecto produce

  • La inserción en MariaDB sí ocurre.
  • La lectura posterior no encuentra filas para el alumno.
  • El repositorio devuelve una lista vacía.
  • La media calculada por el servicio cae a 0.0.

Resultado real del build 3
