Capítulo 2

Cómo se conectan Gitea y Jenkins en este laboratorio

El objetivo no es solo que Jenkins clone un repo. El objetivo es que un git push en Gitea provoque automáticamente un build en Jenkins sobre la última revisión remota.

El flujo real que usa el laboratorio

Secuencia
git push local -> Gitea recibe el commit
                 -> Gitea envía POST al webhook relay
                 -> relay pide a Jenkins que ejecute el job
                 -> Jenkins clona el repo desde Gitea
                 -> Jenkins ejecuta el Jenkinsfile de esa revisión

La razón de introducir el relay es pragmática: así se desacopla el disparo del build del mecanismo de polling interno del plugin Git y se obtiene un comportamiento directo y repetible.

Qué crea `bootstrap-gitea.sh`

  • Usuario alumno en Gitea.
  • Repositorio alumno/jenkins-python-lab.
  • Webhook del repo apuntando a http://webhook-relay:8081/gitea/push.

Qué prepara Jenkins al arrancar

  • Usuarios internos del laboratorio.
  • Job jenkins-python-lab.
  • Definición del job como Pipeline script from SCM.
  • SCM remoto: http://gitea:3000/alumno/jenkins-python-lab.git.
Punto técnico importante. El job no usa un script inline en Jenkins. Usa un `Jenkinsfile` que vive dentro del repositorio remoto. Eso permite que el alumno vea una idea central de CI moderna: el pipeline también está versionado en Git.

Por qué no se dejó la explicación en “usa un webhook y ya”

Problema Qué ocurre Cómo lo resuelve el laboratorio
Disparo poco determinista El polling del plugin Git puede no lanzar el build exactamente cuando el alumno espera. El relay hace un `POST /build` explícito a Jenkins.
Credenciales y CSRF El disparo remoto de Jenkins suele chocar con autenticación y crumb. El laboratorio desactiva CSRF porque el entorno es local y docente.
Visibilidad pedagógica Si el alumno solo ve una URL de webhook, no entiende la cadena completa. La arquitectura del laboratorio enseña cada salto del proceso.
