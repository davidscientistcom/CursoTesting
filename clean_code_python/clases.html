Clases & SOLID

Las clases deben ser pequeñas. Mientras que las funciones se miden en líneas, las clases se miden por responsabilidades. El pilar fundamental es la cohesión profunda entre métodos y atributos.

1. Single Responsibility Principle (SRP)

Una clase debe tener una, y solo una, razón para cambiar. Si puedes pensar en más de una responsabilidad para una clase, está haciendo demasiado y debe ser dividida.

❌ Mala Práctica
class Employee:
    def __init__(self, name, hours):
        self.name = name
        self.hours = hours

    def calculate_pay(self):
        # Business logic
        return self.hours * 20

    def save_to_database(self):
        # Database logic
        pass
        
    def generate_report(self):
        # Presentation logic
        pass
✅ Buena Práctica
class EmployeeData:
    def __init__(self, name, hours):
        self.name = name
        self.hours = hours

class PayCalculator:
    def calculate(self, emp: EmployeeData):
        return emp.hours * 20

class EmployeeRepository:
    def save(self, emp: EmployeeData):
        pass

2. Open/Closed Principle (OCP) en Python

El código debería estar abierto para la extensión pero cerrado para modificación. En Python, aprovechamos el Polimorfismo mediante `Protocolos` (Python 3.8+) o Clases Base Abstractas (ABCs) para evitar tocar la lógica principal (los "Use Cases") cuando agregamos nuevos comportamientos.

❌ Mala Práctica
class NotificationService:
    def send(self, type: str, message: str):
        if type == "email":
            # Send Email
            pass
        elif type == "sms":
            # Send SMS
            pass
        # Si queremos añadir PUSH, 
        # hay que tocar esta clase
✅ Buena Práctica
from typing import Protocol

class Notifier(Protocol):
    def send(self, message: str): ...

class EmailNotifier:
    def send(self, message: str): ...

class SMSNotifier:
    def send(self, message: str): ...

# Esta clase nunca cambiará al añadir
# un nuevo tipo de Notifier
class NotificationService:
    def __init__(self, notifier: Notifier):
        self.notifier = notifier
        
    def alert(self, message: str):
        self.notifier.send(message)

3. Cohesión y División de Clases

Las clases deben tener una alta cohesión: la mayoría de los métodos deberían usar la mayoría de los atributos de instancia. Si encuentras un grupo de métodos que solo usa un subconjunto específico de variables de la clase (`self.a` y `self.b`), seguramente ese grupo quiere ser una clase diferente.

❌ Baja Cohesión
class UserSettingsManager:
    def __init__(self):
        self.theme = "dark"
        self.font = "Arial"
        
        self.db_url = "postgres://..."
        self.cache_host = "redis"

    def apply_ui_settings(self):
        # Usa solo theme y font
        pass

    def connect_infrastructure(self):
        # Usa solo db_url y cache_host
        pass
✅ Alta Cohesión
class UISettings:
    def __init__(self):
        self.theme = "dark"
        self.font = "Arial"
        
    def apply(self):
        pass

class InfraSettings:
    def __init__(self):
        self.db_url = "postgres://..."
        self.cache_host = "redis"
        
    def connect(self):
        pass

4. Ley de Demeter (Tell, Don't Ask)

Un objeto no debe conocer la estructura interna de los objetos con los que interactúa. En lugar de pedirle a un objeto sus partes (getters) y operar en ellas, dile al objeto lo que debe hacer (métodos de acción). En Python, los encadenamientos de métodos `obj.get_a().get_b().do_c()` ("train wrecks") son una clara violación.

❌ Violación (Ask)
# Accediendo a internos profundamente
zip_code = customer.get_address().get_postal_info().get_zip()

# O en Python moderno con propiedades:
zip_code = customer.address.postal.zip
✅ Cumplimiento (Tell)
# Delegando la responsabilidad
zip_code = customer.get_zip_code()

# Y dentro de Customer:
class Customer:
    def get_zip_code(self):
        return self.address.get_zip_code()
⚠️ Evitar Clases Innecesarias en Python: Python no es Java o C#. Si tienes una clase con solo `__init__` y un método que hace algo, esa clase quiere ser una función. Si una clase solo tiene getters y setters sin comportamiento, debería ser una `dataclass` o un `dict`. Modismo Python: Keep it Simple.
