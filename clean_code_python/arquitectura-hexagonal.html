Arquitectura Hexagonal

También conocida como Arquitectura de Puertos y Adaptadores. Su propósito es aislar tu lógica de negocio central (Casos de Uso) de las preocupaciones externas (APIs, Bases de Datos, UI), permitiendo un código altísimamente testeable y mantenible.

1. El Antipatrón: Arquitectura Altamente Acoplada

El problema con la mayoría de scripts y aplicaciones clásicas es que el controlador web, las reglas del negocio, el formato de salida y la base de datos están todos entrelazados. Es imposible hacer tests unitarios puros porque siempre dependes de infraestructura pesada.

❌ Enfoque Tradicional (Mala Práctica)
# user_service.py (Acoplamiento a SQLAlchemy y Flask/FastAPI)
import requests
from database import db_session
from models import UserDB

def register_user(email: str, name: str):
    # Regla de Negocio 1: Validar formato
    if "@" not in email:
        raise ValueError("Invalid email")
        
    # INFRAESTRUCTURA DB
    existing = db_session.query(UserDB).filter_by(email=email).first()
    if existing:
        raise ValueError("User exists")
        
    user = UserDB(email=email, name=name)
    db_session.add(user)
    db_session.commit()
    
    # INFRAESTRUCTURA EXTERNA: Enviar Email (API Rest)
    requests.post("https://api.sendgrid.com/send", json={
        "to": email, 
        "body": f"Welcome {name}!"
    })
    
    return user

¿Por qué esto es un problema crítico para el Testing?

  • Dependencia de BD: Para testear que la regla del `@` funciona, ¡necesitas levantar una base de datos o hacer `mock(db_session)` (que es muy denso y propenso a roturas)!
  • Dependencia de Red: Para probar que no puedes registrar un usuario duplicado, el test intentará enviar un email con la librería `requests`. Te verás forzado a parchear @patch('requests.post') constatemente.
  • Difícil de cambiar: Si mañana migramos a MongoDB, o cambiamos SendGrid por Amazon SES, toda tu lógica central, que debería ser solo reglas de negocio "if user exists", cambia brutalmente.

2. La Solución Hexagonal (Puertos y Adaptadores)

La arquitectura hexagonal define un dominio puro (las Entidades y los Use Cases). Nada en el dominio sabe que existen BBDD o Internet. Para comunicarse con el exterior, el Dominio define Puertos (Interfaces/Protocolos en Python). La infraestructura concreta, como PostgreSQL o AWS, implementa esos Puertos a través de Adaptadores.


Paso 1: El Dominio Puro y los Puertos (Interfaces)

✅ Dominio/Puertos (Ningún import de BD o Red)
# domain/user.py
from dataclasses import dataclass
from typing import Protocol, Optional

@dataclass
class User:
    email: str
    name: str

# PUERTO: Lo que la aplicación necesita para persistir datos.
class UserRepository(Protocol):
    def get_by_email(self, email: str) -> Optional[User]: ...
    def save(self, user: User) -> None: ...

# PUERTO: Lo que la aplicación necesita para notificar.
class EmailSender(Protocol):
    def send_welcome_email(self, email: str, name: str) -> None: ...

Paso 2: El Orquestador (Caso de Uso central)

Aquí reside la Lógica de Negocio. Fíjate que solo se comunica vía constructos genéricos de Python (los Puertos `Protocol`), usando la técnica de Inyección de Dependencias.

✅ Use Case (100% Cubrible por Tests Rápídos)
# application/register_user.py
from domain.user import User, UserRepository, EmailSender

class RegisterUserUseCase:
    # INYECCIÓN: Recibe la infraestructura desde fuera.
    def __init__(self, repo: UserRepository, mailer: EmailSender):
        self.repo = repo
        self.mailer = mailer

    def execute(self, email: str, name: str) -> User:
        if "@" not in email:
            raise ValueError("Invalid email")
            
        if self.repo.get_by_email(email) is not None:
            raise ValueError("User exists")
            
        new_user = User(email=email, name=name)
        
        self.repo.save(new_user)
        self.mailer.send_welcome_email(email, name)
        
        return new_user

Paso 3: La Infraestructura (Los Adaptadores Reales)

Este es el borde exterior del hexágono. Son componentes sucios o pesados de I/O que implementan (se ajustan) al Protocolo definido por el centro (Dominio/Puertos). Son los "Adaptadores".

✅ Adaptadores de Infraestructura
# infra/db_user_repository.py
from sqlalchemy.orm import Session
from domain.user import User

class SQLAlchemyUserRepository: # Implementa UserRepository
    def __init__(self, session: Session):
        self.session = session

    def get_by_email(self, email: str) -> User | None:
        # Lógica sucia específica a SQLAlchemy
        model = self.session.query(UserModel).filter_by(e=email).first()
        return User(email=model.e, name=model.n) if model else None

    def save(self, user: User) -> None:
        model = UserModel(e=user.email, n=user.name)
        self.session.add(model)
        self.session.commit()

# infra/sendgrid_mailer.py
import requests

class SendGridEmailSender: # Implementa EmailSender
    def send_welcome_email(self, email: str, name: str) -> None:
        requests.post("https://api.sendgrid.com", json={"foo":"bar"})

Paso 4: Inicializando todo junto (El "Composition Root")

En el framework que elijas (ej. FastAPI), instancias los adaptadores y se los pasas al Caso de Uso.

✅ composition_root.py (Entrypoint)
# main.py (FastAPI route)
from application.register_user import RegisterUserUseCase
from infra.db_user_repository import SQLAlchemyUserRepository
from infra.sendgrid_mailer import SendGridEmailSender

@app.post("/users")
def api_register_user(email: str, name: str):
    # Instanciamos adaptadores
    repo = SQLAlchemyUserRepository(db_session)
    mailer = SendGridEmailSender()
    
    # Inyectamos en el centro
    use_case = RegisterUserUseCase(repo=repo, mailer=mailer)
    
    # Ejecutamos!
    return use_case.execute(email, name)
🧪 La Magia del Testing Vuela: Para testear el RegisterUserUseCase (la parte más importante del app), ya no tienes que emular base de datos ni `requests`. Tu test unitario es ultra rápido instanciando mocks simples de los protocolos: `class MockRepo: ...` y `class MockMailer: ...` que pasas al constructor en los tests.
