Capítulo 5

Manejo de Errores

El manejo de errores es importante, pero si oscurece la lógica principal de la aplicación, está mal diseñado. "El manejo de errores es una cosa", según el Single Responsibility Principle.

1. Usa Excepciones en lugar de Códigos de Retorno

Retornar flags (como -1, False, o "ERROR") obliga al llamador a revisar los errores inmediatamente después de cada llamada, desordenando totalmente el flujo principal del código. Lanza excepciones.

❌ Mala Práctica (C style)
def delete_page(page):
    if delete_page_record(page) == 'OK':
        if registry.delete_auth() == 'OK':
            logger.log("Registro borrado")
        else:
            logger.log("Falló Auth")
    else:
        logger.log("Falló DB")
✅ Buena Práctica
def delete_page(page):
    try:
        delete_page_record(page)
        registry.delete_auth()
        logger.log("Registro borrado")
    except Exception as e:
        logger.log(f"Falló al borrar: {e}")

2. Extrae los bloques Try/Except

Los bloques `try/except` son feos estructuralmente, confunden su propia estructura iterando el flujo regular en flujos de error. Según Clean Code, extraer el contenido de un try a una función diferente clarifica inmensamente el código.

❌ Try/Except largo
def copy_files(files):
    for f in files:
        try:
            # Mucha lógica de negocio
            # Lectura del file
            # Validación
            # Copiar a otro lado
            # Cerrar archivo...
            pass
        except OSError:
            logger.error("Error OS en archivo")
✅ Buena Práctica
def copy_files(files):
    for f in files:
        _copy_single_file(f)

def _copy_single_file(f):
    try:
        # Pura lógica de negocio, no anidada
        _execute_copy_process(f)
    except OSError:
        logger.error(f"Error {f}")

3. Python: EAFP vs LBYL

La comunidad Python adora el concepto EAFP (Easier to Ask for Forgiveness than Permission) que significa "usa excepciones". Contrasta con LBYL (Look Before You Leap) donde revisas largas cademas de `if not object_is_none:` antes de actuar. Las excepciones son más rápidas y claras en Python.

❌ LBYL (Mala en Python)
def process_user(user_dict):
    # Preguntar permiso
    if 'config' in user_dict:
        config = user_dict['config']
        if 'theme' in config:
            return config['theme']
    return "default"
✅ EAFP (Pythonic)
def process_user(user_dict):
    # Pedir perdón
    try:
        return user_dict['config']['theme']
    except KeyError:
        return "default"

4. ¡No devuelvas ni pases `None`!

Devolver `None` por costumbre obliga a todos los scripts conectables intermedios y de cliente a llenar el programa con bloques if check is not None:. Cuando un error ocurre, lanza una Excepción. Si no hay error, pero hay "ausencia de datos", prefiere usar el Patrón de Objeto Nulo o devolver listas vacías [] en vez de None.

❌ Común (Devolver None)
def get_active_users():
    users = DB.exec("SELECT * FROM users")
    if not users:
        return None
    return users

# Luego quien llama:
users = get_active_users()
if users is not None:
    for u in users: ...
✅ Limpio (Listas vacías)
def get_active_users():
    users = DB.exec("...")
    # si está vacío retorna []
    return users if users else []

# El que llama no necesita ifs
for u in get_active_users():
    # ...
    pass
