Capítulo 2

El Arte de las Funciones

La regla central de las funciones: "Deben ser pequeñas. La segunda regla es que deben ser todavía más pequeñas". Deben hacer *una sola* cosa y hacerla bien.

1. Pequeñas y Una Sola Cosa (SRP)

Las funciones deben contar una historia, paso a paso, leyendo de arriba a abajo. Si una función tiene múltiples niveles de indentación o if/else/for anidados, probablemente está haciendo demasiadas cosas.

❌ Mala Práctica
def process_user_data(user_id):
    # Fetch data
    db = get_db_connection()
    user_data = db.query(f"SELECT * FROM users WHERE id={user_id}")
    if not user_data:
        raise Exception("Not found")

    # Format data
    formatted = {
        "name": user_data['name'].upper(),
        "age": user_data['age']
    }

    # Send Notification
    email_client = EmailClient()
    email_client.send(user_data['email'], "Processed", formatted)
✅ Buena Práctica
def process_user_data(user_id):
    user_data = _fetch_user_from_db(user_id)
    formatted_data = _format_user_data(user_data)
    _notify_user(user_data['email'], formatted_data)

# Las funciones privadas (`_fetch...`) encapsulan sus detalles

2. Evitar Flags en Argumentos (Booleanos)

Pasar un argumento booleano (una 'flag') a una función es una declaración explícita de que la función hace más de una cosa: hace una cosa si es `True` y otra diferente si es `False`.

❌ Mala Práctica
def create_file(filename, is_temp):
    if is_temp:
        path = f"/tmp/{filename}"
    else:
        path = f"/var/lib/{filename}"
    
    with open(path, 'w') as f:
        f.write("test")
✅ Buena Práctica
def create_temp_file(filename):
    path = f"/tmp/{filename}"
    _write_to_path(path, "test")

def create_permanent_file(filename):
    path = f"/var/lib/{filename}"
    _write_to_path(path, "test")

3. Argumentos y Objetos de Datos

El número ideal de argumentos para una función es cero (niladic). Luego uno (monadic), y luego dos (dyadic). Funciones con tres o más argumentos (polyadic) deben evitarse en lo posible. Usa objetos, Dataclasses o NamedTuples.

❌ Mala Práctica
def make_circle(x, y, radius, color, is_filled):
    # Too many args
    pass

make_circle(10.0, 15.5, 5.0, "red", True)
✅ Buena Práctica
@dataclass
class Point:
    x: float
    y: float

@dataclass
class CircleConfig:
    radius: float
    color: str
    is_filled: bool

def make_circle(center: Point, config: CircleConfig):
    pass

4. Command Query Separation (CQS)

Las funciones deberían hacer algo (Command) o responder algo (Query), pero no ambas cosas a la vez.

❌ Mala Práctica
def set_user_property(user, prop, value):
    try:
        user[prop] = value
        return True # Mutó y retornó estado
    except KeyError:
        return False

# Causa confusión: ¿Qué significa "if set_property..."?
if set_user_property(user, 'age', 30):
    pass
✅ Buena Práctica
def user_has_property(user, prop):
    return prop in user

def set_user_property(user, prop, value):
    user[prop] = value

if not user_has_property(user, 'age'):
    set_user_property(user, 'age', 30)
🛡️ Testing Course Hint: "Aislabilidad". Si en un test unitario mockear algo es increíblemente doloroso, es señal de que tu función no está haciendo una sola cosa. Romper la función en funciones más pequeñas puras ("pure functions" que no tocan DB o APIs ni mutean estado global) hará el testing inmensamente más fácil, permitiéndote inyectar dependencias y predecir outputs.
