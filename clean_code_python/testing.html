Tests Limpios y Arquitectura Testeable

El código de producción es solo la mitad del software; la otra mitad son los tests. Los tests unitarios no son un añadido, son la red de seguridad que permite que el diseño (la arquitectura) evolucione sin miedo.

1. Las 3 Leyes del TDD (Test-Driven Development)

El ciclo Red-Green-Refactor se rige por tres estrictas leyes que aseguran que todo el código esté cubierto, que los tests guíen el diseño, y que nunca haya exceso de abstracción.

  • Primera Ley: No puedes escribir código de producción hasta haber escrito un test unitario que falle.
  • Segunda Ley: No puedes escribir más del test unitario que lo suficiente para que falle (no compilar/errores léxicos cuentan como fallo).
  • Tercera Ley: No puedes escribir más código de producción que el suficiente para hacer pasar el test que está fallando actualmente.

2. Inyección de Dependencias (El Secreto de la Testeabilidad)

Si una de tus clases o funciones instancia sus propias dependencias internamente (por ejemplo, una conexión a Base de Datos o a una API de Emails), ese código es extremadamente difícil de testear en aislamiento. La solución es pasarle (inyeactar) esa dependencia ya instanciada.

❌ In-testeable (Alto Acoplamiento)
import requests

class WeatherApp:
    def __init__(self):
        # HARDCODED: Díficil de mockear!
        self.api_url = "http://api.weather"
        
    def get_temp(self, city):
        # Llama a red real cada vez!
        res = requests.get(f"{self.api_url}/{city}")
        return res.json()['temp']
✅ Testeable (Inyección)
from typing import Protocol

class HttpClient(Protocol):
    def get(self, url: str) -> dict: ...

class WeatherApp:
    # Inyectamos el cliente!
    def __init__(self, client: HttpClient):
        self.client = client
        
    def get_temp(self, city):
        res = self.client.get(f"url/{city}")
        return res['temp']

# En Test, pasamos un MockHttpClient !

3. Separar Lógica Pura de Efectos Secundarios (I/O)

Gran parte de tu sistema no debería necesitar "Mocks" complejos. Si aíslas la lógica de negocio central (matemáticas, reglas de usuario, transformaciones de datos) de los efectos secundarios (llamar bases de datos, leer archivos, pintar UI), puedes testear el núcleo de manera 100% aislada, super rápido.

❌ Mezclado (Lógica + I/O)
def pay_user(user_id):
    # I/O Database
    user = db.query(f"SELECT * FROM u WHERE id={user_id}")
    
    # Lógica
    amount = user.hours * 20
    if user.is_senior:
        amount += 100 
        
    # I/O API Externa
    payment_api.send(user.account, amount)
✅ Separado (Functional Core)
# 1. Función Pura (Fácil de testear!)
def calculate_pay(user: dict) -> int:
    amount = user['hours'] * 20
    if user['is_senior']:
        amount += 100
    return amount

# 2. Orquestador Imperativo (Mocks aquí)
def pay_user(user_id, repo, payment_gateway):
    user = repo.get_user(user_id)
    amount = calculate_pay(user)
    payment_gateway.send(user.account, amount)

4. Reglas F.I.R.S.T para Tests Limpios

Tus tests de código (pytest, unittest) deben seguir unacrónimo fundamental para que no se conviertan en una carga a largo plazo:

  • F (Fast): Rápidos. Si los tests tardan minutos, nadie los correrá. Un test unitario debe ejecutarse en milisegundos.
  • I (Independent): Independientes. Ningún test debe depender del estado dejado por otro test. Deben poder correr en paralelo o desorden.
  • R (Repeatable): Repetibles. Deben dar el mismo resultado local, en la CI (GitHub Actions) y sin red.
  • S (Self-Validating): Auto-validables. Salida binaria: pasan o no pasan. No debes tener que leer un log o un TXT para saber si pasaron.
  • T (Timely): Oportunos. En TDD, los tests se escriben justo antes del código de producción que los hace pasar.

5. Patrón AAA (Arrange, Act, Assert)

Cada test unitario debe tener una estructura visual clara separada por un salto de línea.

def test_calculate_pay_for_senior_employee_includes_bonus():
    # Arrange (Preparar datos y pre-condiciones)
    senior_employee = {"hours": 10, "is_senior": True}
    
    # Act (Llamar a la unidad, la acción real a evaluar)
    pay = calculate_pay(senior_employee)
    
    # Assert (Comprobar resultados)
    assert pay == 300  # (10 * 20) + 100 bonus
