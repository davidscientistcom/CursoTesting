Comentarios & Formato

Un comentario es, muchas veces, un fracaso de nuestra capacidad de expresarnos en el código. El código miente y cambia, el comentario miente más tiempo. Y sobre el formato: deja que las herramientas automáticas hablen.

1. Explícate en el Código

Antes de escribir un comentario explicativo, pregúntate si no puedes simplemente extraer esa lógica a una función con un nombre descriptivo o renombrar las variables implicadas.

❌ Mala Práctica
# Comprobar si el empleado es elegible
# para recibir beneficios este año completo
if (employee.flags & HOURLY_FLAG) and \
   (employee.age > 65):
    # Logica beneficios
    pass
✅ Buena Práctica

if employee.is_eligible_for_full_benefits():
    _give_benefits()
        
# 'is_eligible_for_full_benefits' se
# explica por sí mismo

2. Malos Comentarios: Ruido y Diarios

Comentarios que declaran lo obvio, o bloques gigantes al inicio del archivo con un "Diario de Cambios" (Changelog) son historia del pasado. Hoy tenemos herramientas como Git para registrar quién hizo qué y cuándo.

❌ Mala Práctica
"""
12/03/2026 - Bob - Added is_active flag
13/03/2026 - Alice - Fixed bug in active
"""
def is_active():
    # Retorna si es activo
    return self._active

✅ Buena Práctica
def es_activo():
    return self._activo

3. Docstrings vs Comentarios

En Python, docstrings (documentación en strings de triple interlineado) son distintos del comentario de línea (#). Docstrings se evalúan como metadatos en __doc__ y son la manera estandar de documentar Módulos, Clases y Funciones API públicas. No deberías forzar un docstring si el nombre de la función dice todo, salvo en código Open Source o de librería (APIs públicas).

4. Formateo y PEP 8

El formato visual es un proceso de comunicación. Las reglas estándar hoy en Python implican usar el estándar PEP 8 a través de herramientas automatizadas en los pipelines o el IDE local. Nunca corrijas el formato a mano.

  • Herramientas recomendadas: Black ("The uncompromising formatter"), Ruff, o Blue.
  • Densidad Vertical: Líneas conceptualmente afines deben estar sin espacios. Separa conceptos diferentes (o flujos) con una línea en blanco (Metáfora de párrafos en periódico).
  • Proximidad Horizontal: Variables se declaran lo más cerca posible de su uso inicial, y no listadas en el inicio de la función "estilo C".
❌ Estilo desordenado
def _process(self,data):
    x=0
    a,b,c=None,None,None
    for i in data:
         if i>0:
             x+=i
    return x
✅ PEP 8 c/ Black format
def _process(self, data: list[int]) -> int:
    total_sum = 0
    
    for item in data:
        if item > 0:
            total_sum += item
            
    return total_sum
