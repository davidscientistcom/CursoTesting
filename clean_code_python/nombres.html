Capítulo 1

Nombres Significativos

El primer paso hacia el código limpio es dar nombres que revelen intención. Si necesitas un comentario para explicar qué hace una variable, el nombre es incorrecto.

1. Usa Nombres que Revelen la Intención

El nombre de una variable, función o clase debe responder a las grandes preguntas: por qué existe, qué hace y cómo se usa. Si necesitas un comentario, el nombre no revela intención.

❌ Mala Práctica
# Tiempo transcurrido en días
d = 0  

# ¿Qué es lst? ¿Qué es 4?
def get_them(lst):
    list1 = []
    for x in lst:
        if x[0] == 4:
            list1.append(x)
    return list1
✅ Buena Práctica
elapsed_time_in_days = 0

def get_flagged_cells(game_board):
    flagged_cells = []
    for cell in game_board:
        if cell.is_flagged():
            flagged_cells.append(cell)
    return flagged_cells

2. Evita la Desinformación

Los programadores deben evitar dejar pistas falsas que oscurezcan el significado del código. Evita las palabras cuyos significados varían del pretendido.

❌ Mala Práctica
# Es engañoso si no es de tipo List
account_list = set() 

# Nombres que varían por mínimos caracteres
xyz_controller_for_efficient_handling_of_strings = ...
xyz_controller_for_efficient_storage_of_strings = ...

# Usar 'l' (ele minúscula) o 'O' (o mayúscula)
O = 1
l = 2
if O == l:
    a = l1
✅ Buena Práctica
account_group = set()
accounts = set()

# Nombres fácilmente distinguibles
string_handler_controller = ...
string_storage_controller = ...

# Evita identificadores confusos
zero = 1
two = 2
if zero == two:
    a = limit_1

3. Haz Distinciones Significativas

Evita agregar números o palabras redundantes ("noise words") a las variables. Si los nombres deben ser distintos, también deben significar cosas diferentes.

❌ Mala Práctica
def copy_chars(a1, a2):
    for i in range(len(a1)):
        a2[i] = a1[i]

class ProductInfo: ...
class ProductData: ...

# Tipos en los nombres (Hungarian Notation)
name_string = "Juan"
✅ Buena Práctica
def copy_chars(source, destination):
    for index in range(len(source)):
        destination[index] = source[index]

class Product: ...
class ProductSnapshot: ...

# Tipado implícito (usa type hints mejor)
name: str = "Juan"

4. Usa Nombres Pronunciables y Buscables

Gran parte de la programación se trata de leer código en equipo. Si no puedes pronunciarlo o buscarlo fácilmente, estás obstaculizando el proceso.

❌ Mala Práctica
class DtaRcrd102:
    def __init__(self):
        self.genymdhms = current_time()
        self.modymdhms = current_time()
        self.pszqint = "102"
✅ Buena Práctica
class Customer:
    def __init__(self):
        self.generation_timestamp = current_time()
        self.modification_timestamp = current_time()
        self.record_id = "102"

5. Estándar PEP 8 en Python

El Clean Code es lenguaje agnóstico, pero la implementación sí importa el lenguaje. En Python, el documento pep-0008 dicta las reglas base para nombres.

  • Módulos y paquetes: snake_case (cortos, sin guiones bajos si es posible).
  • Clases: PascalCase (ej. MyClass). Excepciones para clases internas/tooling: `_MyClass`.
  • Funciones y Variables: snake_case (ej. process_data).
  • Constantes: SCREAMING_SNAKE_CASE (ej. MAX_OVERFLOW).
  • Privacidad de métodos/atributos: Prefijo de guión bajo (_private_method) en clases.
💡 Tip de Arquitectura de Testing: Un buen nombrado facilita enormemente los tests. Un test que valide `test_calculate_pay_for_hourly_employee()` es mucho más entendible y confiable que simplemente `test_calc1()`.
