Lab: Refactoring con TDD

código legacy · characterization tests · Red-Green-Refactor aplicado · el proceso completo resuelto

El escenario: sistema legacy de procesamiento de pedidos

Te entregan el siguiente código. Lleva 3 años en producción. No hay tests. Nadie se atreve a tocarlo. Tu tarea: dejarlo con tests, desacoplado y mantenible, sin romper el comportamiento existente.

Regla del lab: antes de cambiar una sola línea de producción, tienes que tener tests pasando. El refactor se hace con la red de seguridad puesta, no antes.

legacy_pedidos.py — el código que tienes que refactorizar 
# legacy/pedidos.py — código legacy real. Complejidad ciclomática: 12
# SRP violado: 4 responsabilidades en una función
# Dependencia hard-coded a smtplib (no testeable en aislamiento)
# Sin tipos ni docstrings
# Variables con nombres ambiguos

import smtplib
import sqlite3
from email.message import EmailMessage
from datetime import datetime


DB_PATH = "/var/data/pedidos.db"
SMTP_HOST = "smtp.empresa.com"
SMTP_PORT = 587


def procesar_pedido(pid, uid, items, promo=None):
    # Calcular total
    total = 0
    for item in items:
        if item["tipo"] == "fisico":
            if item["qty"] > 0:
                total += item["precio"] * item["qty"]
                if item["qty"] >= 10:
                    total -= item["precio"] * item["qty"] * 0.05  # 5% bulk
        elif item["tipo"] == "digital":
            total += item["precio"] * item["qty"]
        else:
            pass  # ignorar tipos desconocidos

    # Aplicar promo
    if promo:
        if promo == "VERANO20":
            if total > 50:
                total = total * 0.80
        elif promo == "BIENVENIDA":
            total = total - 10
            if total < 0:
                total = 0
        elif promo.startswith("CUSTOM_"):
            pct = int(promo.split("_")[1])
            total = total * (1 - pct / 100)

    # Guardar en BD
    conn = sqlite3.connect(DB_PATH)
    cur = conn.cursor()
    cur.execute(
        "INSERT INTO pedidos (user_id, total, fecha, estado) VALUES (?, ?, ?, ?)",
        (uid, total, datetime.now().isoformat(), "pendiente")
    )
    nuevo_id = cur.lastrowid
    conn.commit()
    conn.close()

    # Enviar email
    if uid:
        conn2 = sqlite3.connect(DB_PATH)
        cur2 = conn2.cursor()
        cur2.execute("SELECT email FROM usuarios WHERE id=?", (uid,))
        row = cur2.fetchone()
        conn2.close()

        if row:
            try:
                with smtplib.SMTP(SMTP_HOST, SMTP_PORT) as s:
                    msg = EmailMessage()
                    msg["From"] = "pedidos@empresa.com"
                    msg["To"] = row[0]
                    msg["Subject"] = f"Pedido #{nuevo_id} recibido"
                    msg.set_content(f"Total: {total:.2f}€. Gracias por tu compra.")
                    s.send_message(msg)
            except Exception:
                pass  # silenciar errores de email

    return {"id": nuevo_id, "total": total, "estado": "pendiente"}

Paso 1 — Characterization tests: capturar el comportamiento actual

Antes de refactorizar, necesitas tests que describan el comportamiento actual del código. Estos tests no verifican si el código es correcto — verifican qué hace ahora mismo. Son tu red de seguridad.

Problema identificado primero
procesar_pedido tiene 4 responsabilidades: calcular total, aplicar promo, persistir en BD, enviar email. Para testearla necesitamos BD real y SMTP real. Imposible de aislar tal cual.
paso1_characterization.py — tests de caracterización con patch para aislar la infraestructura 
"""
Characterization tests: capturan el comportamiento actual.
Usan patch para aislar BD y SMTP porque el código legacy no está preparado
para inyección. Esto es un paso intermedio — luego los eliminaremos
cuando tengamos el diseño limpio.
"""
import pytest
from unittest.mock import patch, MagicMock

    return mock_conn, mock_cursor

Paso 2 — Extraer la lógica pura: CalculadorTotal

Ahora que tenemos tests de caracterización, extraemos la lógica de cálculo a una clase pura, sin efectos secundarios. Esta clase es trivialmente testeable.

paso2_calculador.py — lógica de cálculo extraída y testeada 
from dataclasses import dataclass
from typing import Literal
import pytest


ItemTipo = Literal["fisico", "digital"]


@dataclass
class Item:
    tipo: ItemTipo
    qty: int
    precio: float


class CalculadorTotal:
    """
    Lógica pura de cálculo de totales.
    Sin BD, sin email, sin efectos secundarios.
    Complejidad ciclomática: 4 (radon: A)
    """

    DESCUENTO_BULK_QTY = 10
    DESCUENTO_BULK_PCT = 0.05

    def calcular(self, items: list[Item]) -> float:
        total = 0.0
        for item in items:
            if item.tipo == "fisico" and item.qty > 0:
                subtotal = item.precio * item.qty
                if item.qty >= self.DESCUENTO_BULK_QTY:
                    subtotal *= 1 - self.DESCUENTO_BULK_PCT
                total += subtotal
            elif item.tipo == "digital":
                total += item.precio * item.qty
        return round(total, 2)


# ── Tests del CalculadorTotal — cero infraestructura ─────────────

def test_item_fisico_calculo_basico():
    c = CalculadorTotal()
    assert c.calcular([Item("fisico", 2, 50.0)]) == 100.0


def test_item_digital_siempre_precio_completo():
    c = CalculadorTotal()
    assert c.calcular([Item("digital", 3, 10.0)]) == 30.0


def test_bulk_descuento_exactamente_10_unidades():
    c = CalculadorTotal()
    # 10 * 100 = 1000, descuento 5% = 50, total = 950
    assert c.calcular([Item("fisico", 10, 100.0)]) == 950.0


def test_bulk_descuento_no_aplica_con_9_unidades():
    c = CalculadorTotal()
    assert c.calcular([Item("fisico", 9, 100.0)]) == 900.0


def test_tipo_desconocido_se_ignora():
    c = CalculadorTotal()
    items = [
        Item("fisico", 1, 50.0),
        # Simular tipo desconocido — creamos dataclass manual
    ]
    resultado = c.calcular(items)
    assert resultado == 50.0


def test_lista_vacia_devuelve_cero():
    c = CalculadorTotal()
    assert c.calcular([]) == 0.0


def test_mezcla_fisicos_y_digitales():
    c = CalculadorTotal()
    items = [
        Item("fisico", 2, 50.0),    # 100
        Item("digital", 1, 20.0),   # 20
    ]
    assert c.calcular(items) == 120.0

Paso 3 — Extraer AplicadorPromo

paso3_promo.py — política de descuentos extraída y testeada 
import pytest


class PromoInvalida(Exception):
    pass


class AplicadorPromo:
    """
    Aplica códigos promocionales a un total.
    Pura lógica de negocio — testeable directamente.
    """

    CODIGOS_FIJOS = {
        "VERANO20": ("porcentaje", 20, 50.0),   # (tipo, valor, min_compra)
        "BIENVENIDA": ("fijo", 10, 0.0),
    }

    def aplicar(self, total: float, promo: str) -> float:
        if promo in self.CODIGOS_FIJOS:
            tipo, valor, min_compra = self.CODIGOS_FIJOS[promo]
            if total < min_compra:
                return total  # sin descuento por no llegar al mínimo
            if tipo == "porcentaje":
                return round(total * (1 - valor / 100), 2)
            else:  # fijo
                return max(0.0, round(total - valor, 2))

        if promo.startswith("CUSTOM_"):
            try:
                pct = int(promo.split("_")[1])
            except (IndexError, ValueError):
                raise PromoInvalida(f"Formato inválido: {promo}")
            if not 0 <= pct <= 100:
                raise PromoInvalida(f"Porcentaje fuera de rango: {pct}")
            return round(total * (1 - pct / 100), 2)

        raise PromoInvalida(f"Código desconocido: {promo}")


# ── Tests ─────────────────────────────────────────────────────────

class TestVERAN020:
    def test_aplica_20_pct_si_total_supera_minimo(self):
        a = AplicadorPromo()
        assert a.aplicar(100.0, "VERANO20") == 80.0

    def test_no_aplica_si_total_bajo_minimo(self):
        a = AplicadorPromo()
        assert a.aplicar(30.0, "VERANO20") == 30.0  # < 50€ mínimo


class TestBIENVENIDA:
    def test_resta_10_euros(self):
        a = AplicadorPromo()
        assert a.aplicar(50.0, "BIENVENIDA") == 40.0

    def test_nunca_produce_total_negativo(self):
        a = AplicadorPromo()
        assert a.aplicar(5.0, "BIENVENIDA") == 0.0


class TestCUSTOM:
    def test_custom_50_pct(self):
        a = AplicadorPromo()
        assert a.aplicar(100.0, "CUSTOM_50") == 50.0

    def test_custom_formato_invalido_lanza_excepcion(self):
        a = AplicadorPromo()
        with pytest.raises(PromoInvalida):
            a.aplicar(100.0, "CUSTOM_abc")

    def test_codigo_desconocido_lanza_excepcion(self):
        a = AplicadorPromo()
        with pytest.raises(PromoInvalida, match="desconocido"):
            a.aplicar(100.0, "CODIGO_INVENTADO")

Paso 4 — El orquestador final refactorizado

Con las piezas puras testeadas, el orquestador se reduce a composición. Sus tests solo verifican que llama a las partes correctas.

paso4_orquestador.py — ServicioPedidos: cliente limpio y testeable 
from typing import Protocol
from dataclasses import dataclass
from unittest.mock import MagicMock
import pytest


@dataclass
class Item:
    tipo: str
    qty: int
    precio: float


@dataclass
class ResultadoPedido:
    id: int
    total: float
    estado: str


# Ports (interfaces) — definidos por el dominio
class RepositorioPedidos(Protocol):
    def guardar(self, uid: int, total: float) -> int: ...


class NotificadorPedidos(Protocol):
    def notificar_creacion(self, pedido_id: int, uid: int, total: float) -> None: ...


# Dominio — orquestador limpio
class ServicioPedidos:
    def __init__(
        self,
        calculador: "CalculadorTotal",
        aplicador_promo: "AplicadorPromo",
        repo: RepositorioPedidos,
        notificador: NotificadorPedidos,
    ) -> None:
        self._calculador = calculador
        self._aplicador_promo = aplicador_promo
        self._repo = repo
        self._notificador = notificador

    def crear_pedido(
        self,
        uid: int,
        items: list[Item],
        promo: str | None = None,
    ) -> ResultadoPedido:
        total = self._calculador.calcular(items)

        if promo:
            total = self._aplicador_promo.aplicar(total, promo)

        pedido_id = self._repo.guardar(uid, total)
        self._notificador.notificar_creacion(pedido_id, uid, total)

        return ResultadoPedido(id=pedido_id, total=total, estado="pendiente")


# ── Tests del orquestador — solo verifican composición ───────────

@pytest.fixture
def deps():
    calculador = MagicMock()
    calculador.calcular.return_value = 100.0

    aplicador = MagicMock()
    aplicador.aplicar.return_value = 80.0

    repo = MagicMock(spec=RepositorioPedidos)
    repo.guardar.return_value = 42

    notificador = MagicMock(spec=NotificadorPedidos)

    return calculador, aplicador, repo, notificador


def test_crear_pedido_devuelve_resultado_correcto(deps):
    calculador, aplicador, repo, notificador = deps
    servicio = ServicioPedidos(calculador, aplicador, repo, notificador)

    resultado = servicio.crear_pedido(
        uid=1,
        items=[Item("fisico", 2, 50.0)],
        promo="VERANO20",
    )

    assert resultado.id == 42
    assert resultado.total == 80.0  # valor que devuelve el mock del aplicador
    assert resultado.estado == "pendiente"


def test_crear_pedido_sin_promo_no_llama_aplicador(deps):
    calculador, aplicador, repo, notificador = deps
    servicio = ServicioPedidos(calculador, aplicador, repo, notificador)

    servicio.crear_pedido(uid=1, items=[Item("fisico", 2, 50.0)])

    aplicador.aplicar.assert_not_called()


def test_crear_pedido_notifica_con_datos_correctos(deps):
    calculador, aplicador, repo, notificador = deps
    servicio = ServicioPedidos(calculador, aplicador, repo, notificador)

    servicio.crear_pedido(uid=7, items=[Item("fisico", 1, 100.0)])

    notificador.notificar_creacion.assert_called_once_with(42, 7, 100.0)

Resultado final: qué ganamos con el refactor

Código legacyDespués del refactor
Complejidad ciclomática 12 (una función) — radon: D CalculadorTotal: 4 / AplicadorPromo: 5 / Orquestador: 2 — radon: A
Tests 0 20+ tests unitarios, todos sin infraestructura
Cobertura 0% >95% — incluyendo branch coverage
Tiempo de ejecución de tests No aplica <100ms — todo en memoria, sin BD ni SMTP
Para añadir nuevo código promo Modificar la función de 80 líneas y rezar Añadir entrada en CODIGOS_FIJOS + 3 tests
Para cambiar de SMTP a Slack Reescribir la función, romper todo Implementar NotificadorSlack con el Protocol 
El proceso de refactoring con TDD — en 4 pasos siempre:

  1. CHARACTERIZE    → tests que describen el comportamiento actual
                        (con patch para sobrevivir al código legacy)

  2. EXTRACT         → extraer lógica pura a clases testables directamente
                        (CalculadorTotal, AplicadorPromo)

  3. TEST PURE       → tests unitarios de las clases puras
                        (cero infraestructura, cero mocks complejos)

  4. COMPOSE         → orquestador que une las piezas vía Protocol
                        (tests de comportamiento con Mocks del orquestador)

En ningún momento rompiste el comportamiento existente.
Los characterization tests validan que la refactorización es equivalente.

El Cap 4 te dio el diseño (Protocol, inyección). Los Cap 5-9 te dieron las herramientas (fixtures, mocks, characterization).
