Coverage y calidad de código

coverage.py · pytest-cov · ruff · mypy · black · bandit · el pipeline completo

Coverage: qué mide y qué no mide

La cobertura mide qué líneas (o ramas) de código se ejecutaron durante los tests. Es una métrica necesaria pero no suficiente. Un 90% de cobertura no significa que el 90% del comportamiento esté verificado — significa que el 90% del código se ejecutó.

Tipo de coberturaQué mideCuando usarlo
Line coverage Si cada línea fue ejecutada al menos una vez Mínimo indispensable — detecta código muerto
Branch coverage Si cada rama de cada condicional (if/else, try/except) fue ejecutada Recomendado — detecta ramas no testadas en if complejos
Mutation testing Si cambiar el código hace fallar los tests (mutmut, cosmic-ray) Para código crítico — mide calidad real de assertions
WARN La cobertura que engaña: un test que llama a una función sin hacer ningún assert da 100% de cobertura de línea con 0% de valor. El número en sí no garantiza nada — garantiza un mínimo de ejercicio del código, no la correctitud de la verificación.

coverage.py y pytest-cov

coverage_basico.sh — comandos esenciales 
pip install pytest-cov coverage

# Cobertura básica — muestra resumen en terminal
pytest --cov=src

# Muestra líneas no cubiertas (imprescindible)
pytest --cov=src --cov-report=term-missing

# Branch coverage — detecta ramas de if/else no testadas
pytest --cov=src --cov-report=term-missing --cov-branch

# Génerar reporte HTML navegable
pytest --cov=src --cov-report=html
# Abre htmlcov/index.html en el navegador

# Fallar si la cobertura cae del umbral (para CI)
pytest --cov=src --cov-fail-under=80

# Combinar varios formatos
pytest --cov=src \
       --cov-report=term-missing \
       --cov-report=html \
       --cov-report=xml \        # para SonarQube / CI
       --cov-branch \
       --cov-fail-under=80

pragma_ejemplo.py — cuándo usar pragma: no cover 
from typing import TYPE_CHECKING

if TYPE_CHECKING:  # pragma: no cover
    # Imports solo para type checkers, nunca se ejecutan en runtime
    from collections.abc import Iterator


class Servicio:
    def __repr__(self) -> str:  # pragma: no cover
        # No vale la pena testear representaciones de debug
        return f"Servicio(id={self.id})"

    def proceso_imposible_de_alcanzar_en_dominio_normal(self) -> None:
        # Invariante de dominio que nunca debe romperse
        raise NotImplementedError("Este caso no debe ocurrir nunca")  # pragma: no cover

ruff — linter y formatter ultrarrápido

ruff (Astral, 2022) es un linter y formatter escrito en Rust que reemplaza a flake8 + isort + pyupgrade + bandit-lite + black en un solo comando, siendo 10-100x más rápido. Es el estándar emergente para proyectos nuevos en 2024-2026.

ruff_uso.sh — instalación y comandos 
pip install ruff

# Verificar problemas sin modificar
ruff check src/

# Corregir automáticamente lo que puede
ruff check --fix src/

# Formatear código (equivalente a black)
ruff format src/

# Verificar formato sin modificar (para CI)
ruff format --check src/

# Ver todas las reglas violadas con descripción
ruff check src/ --show-fixes

# Uso típico en pre-commit (ver Cap 10):
# ruff check --fix .
# ruff format .
pyproject.toml — configuración de ruff 
[tool.ruff]
target-version = "py311"
line-length = 100

[tool.ruff.lint]
select = [
    "E",    # pycodestyle errors
    "W",    # pycodestyle warnings
    "F",    # pyflakes (imports no usados, variables no definidas)
    "I",    # isort (orden de imports)
    "N",    # pep8-naming
    "UP",   # pyupgrade (modernizar sintaxis: f-strings, type hints)
    "B",    # flake8-bugbear (patrones que suelen ser bugs)
    "S",    # flake8-bandit (seguridad básica)
    "C4",   # flake8-comprehensions
    "SIM",  # flake8-simplify
]
ignore = [
    "S101",  # uso de assert (legítimo en tests)
    "B008",  # llamadas a funciones en argumentos por defecto (FastAPI pattern)
]

[tool.ruff.lint.per-file-ignores]
"tests/*" = ["S101"]  # assert está bien en tests

[tool.ruff.format]
quote-style = "double"
indent-style = "space"
ruff_ejemplos_fixes.py — qué detecta y corrige ruff automáticamente 
# ── Antes de ruff ──────────────────────────────────────────────
import os, sys                    # E401: multiple imports
import json                       # F401: unused import
from typing import List, Dict     # UP006: use list/dict directly (py3.9+)

class usuario:                    # N801: class name should be CapWords
    ID = "admin"

def get_users() -> List[Dict]:    # UP006: modernizar type hints
    items = [x for x in range(10) if x % 2 == 0]
    return items


# ── Después de ruff --fix + ruff format ────────────────────────
import os
import sys

class Usuario:                    # N801 corregido
    ID = "admin"

def get_users() -> list[dict]:    # UP006 corregido
    return [x for x in range(10) if x % 2 == 0]


# ── Detecciones de seguridad (reglas S) ─────────────────────────
import hashlib

# S324: uso de MD5/SHA1 para hashing de contraseñas → inseguro
hash_inseguro = hashlib.md5(b"password").hexdigest()  # ruff lo detecta

# S106: posible contraseña hardcoded
password = "supersecret123"  # ruff alerta

# Forma segura:
import secrets
token = secrets.token_hex(32)     # OK

mypy — tipado estático

mypy analiza las anotaciones de tipo de Python y detecta errores de tipo antes de ejecutar el código. No es obligatorio, pero en proyectos que maduran hacia producción el tipado estático detecta una categoría entera de bugs que los tests no cubren: tipos incorrectos en interfaces.

mypy_uso.sh — instalación y configuración 
pip install mypy

# Verificar un módulo
mypy src/auth.py

# Verificar todo el proyecto
mypy src/

# Strict mode — los errores de tipo nunca pasan desapercibidos
mypy --strict src/
mypy_config.toml — configuración en pyproject.toml 
[tool.mypy]
python_version = "3.11"
strict = true               # activa todas las verificaciones

# Si strict es muy agresivo para proyectos legacy, activar progresivamente:
# disallow_untyped_defs = true     → todas las funciones deben tener tipos
# disallow_any_generics = true     → no List sin parámetro
# warn_return_any = true           → no retornar Any implícito
# warn_unused_ignores = true       → avisar si type: ignore ya no es necesario

ignore_missing_imports = true    # para libs sin stubs (boto3, etc.)

[[tool.mypy.overrides]]
module = "tests.*"
disallow_untyped_defs = false    # tests pueden ser menos estrictos
mypy_ejemplos.py — errores que mypy detecta antes de ejecutar 
from dataclasses import dataclass


@dataclass
class Pedido:
    id: int
    total: float
    estado: str


def aplicar_descuento(pedido: Pedido, porcentaje: float) -> Pedido:
    nuevo_total = pedido.total * (1 - porcentaje / 100)
    return Pedido(id=pedido.id, total=nuevo_total, estado=pedido.estado)


# ── Errores que mypy detecta (no pytest) ───────────────────────

# Error 1: tipo incorrecto en argumento
pedido = Pedido(id=1, total=100.0, estado="pendiente")
resultado = aplicar_descuento(pedido, "10")  # mypy: Argument 2 has incompatible type "str"

# Error 2: atributo que no existe
pedido.precio_final  # mypy: "Pedido" has no attribute "precio_final"

# Error 3: retorno incorrecto
def obtener_estado(pedido: Pedido | None) -> str:
    if pedido is None:
        return None  # mypy: Incompatible return value type (got "None", expected "str")
    return pedido.estado


# ── Forma correcta ──────────────────────────────────────────────
def obtener_estado_correcto(pedido: Pedido | None) -> str | None:
    if pedido is None:
        return None
    return pedido.estado
INFO mypy y pytest se complementan: mypy detecta errores de tipo en interfaces (contratos de funciones), pytest detecta errores de lógica (comportamiento). Juntos cubren dos dimensiones ortogonales de calidad. En proyectos grandes, mypy paga especialmente cuando hay múltiples personas trabajando en el mismo codebase.

bandit — análisis de seguridad

bandit escanea el código Python en busca de patrones de seguridad conocidos: inyecciones SQL, uso de MD5/SHA1 para hashing seguro, subprocesos con shell=True, datos de usuario en consultas dinámicas.

bandit_uso.sh — escaneo de seguridad 
pip install bandit

# Escanear directorio
bandit -r src/

# Solo niveles altos (HIGH) — para CI estricto
bandit -r src/ -ll

# Reporte en JSON para integración con CI
bandit -r src/ -f json -o bandit-report.json

# Excluir tests del escaneo (los tests usan assert, subprocess en entornos controlados)
bandit -r src/ --exclude tests/
bandit_ejemplos.py — qué detecta bandit 
import subprocess
import hashlib
import sqlite3
import pickle


# B602: subprocess con shell=True — inyección de comandos
def ejecutar_backup(nombre_db: str) -> None:
    subprocess.run(f"pg_dump {nombre_db}", shell=True)  # MALO: bandit HIGH
    # Si nombre_db viene del usuario: "mi_db; rm -rf /"
    subprocess.run(["pg_dump", nombre_db], check=True)  # BIEN: lista de args


# B324: MD5 para hashing seguro — criptográficamente débil
def hash_password_malo(password: str) -> str:
    return hashlib.md5(password.encode()).hexdigest()  # MALO: bandit HIGH

import bcrypt
def hash_password_correcto(password: str) -> bytes:
    return bcrypt.hashpw(password.encode(), bcrypt.gensalt())  # BIEN


# B608: posible SQL injection
def buscar_usuario_malo(db: sqlite3.Connection, nombre: str):
    # MALO: inyección si nombre viene del usuario
    db.execute(f"SELECT * FROM usuarios WHERE nombre = '{nombre}'")  # bandit MEDIUM

def buscar_usuario_correcto(db: sqlite3.Connection, nombre: str):
    # BIEN: parámetros enlazados, no interpolación
    db.execute("SELECT * FROM usuarios WHERE nombre = ?", (nombre,))


# B301: pickle con datos no confiables — deserialización arbitraria
def cargar_datos_malo(datos_bytes: bytes):
    return pickle.loads(datos_bytes)  # MALO: ejecución de código arbitrario

import json
def cargar_datos_correcto(datos_str: str):
    return json.loads(datos_str)  # BIEN para datos serializados de forma segura

El pipeline completo de calidad

Las herramientas aisladas tienen valor limitado. El valor real emerge cuando se encadenan en un pipeline que se ejecuta en cada commit (Cap 10). Esta es la configuración estándar:

quality_pipeline.sh — pipeline completo de calidad local y CI 
#!/bin/bash
# scripts/check_quality.sh — ejecutar todo el pipeline de calidad

set -e  # parar al primer error

echo "=== 1. Formato y linting (ruff) ==="
ruff format --check src/ tests/
ruff check src/ tests/

echo "=== 2. Tipos estáticos (mypy) ==="
mypy src/

echo "=== 3. Seguridad (bandit) ==="
bandit -r src/ --exclude tests/ -ll  # solo HIGH y MEDIUM

echo "=== 4. Tests con coverage ==="
pytest \
    --cov=src \
    --cov-report=term-missing \
    --cov-branch \
    --cov-fail-under=80 \
    -v

echo "=== Calidad OK ==="
pyproject.toml — configuración unificada de todas las herramientas 
[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.backends.legacy:build"

[project]
name = "mi-proyecto"
version = "1.0.0"
requires-python = ">=3.11"

[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-v --tb=short --strict-markers"
markers = [
    "integration: requiere infraestructura",
    "slow: tarda más de 5s",
    "smoke: test de humo rápido",
]

[tool.coverage.run]
source = ["src"]
branch = true
omit = ["src/migrations/*"]

[tool.coverage.report]
fail_under = 80
show_missing = true

[tool.ruff]
target-version = "py311"
line-length = 100

[tool.ruff.lint]
select = ["E", "W", "F", "I", "N", "UP", "B", "S", "C4", "SIM"]
ignore = ["S101"]

[tool.mypy]
python_version = "3.11"
strict = true
ignore_missing_imports = true

Pipeline de calidad — cuándo ejecutar cada herramienta:

  Developer commit
       │
       ▼
  pre-commit hook (local, <10s)
  ┌────────────────────────────────┐
  │ ruff format --check            │  → formato consistente
  │ ruff check                     │  → linting rápido
  └────────────────────────────────┘
       │
       ▼
  push / PR (CI, <4min)
  ┌────────────────────────────────┐
  │ mypy --strict                  │  → tipos estáticos
  │ bandit -r src/ -ll             │  → seguridad
  │ pytest --cov --cov-branch      │  → tests + cobertura
  │ pytest --cov-fail-under=80     │  → gate de cobertura
  └────────────────────────────────┘
       │
       ▼
  Deploy (solo si todo pasa)
SUCCESS Separar el pipeline en dos fases es clave: el hook local es rápido (formato + linting, <10s) para no bloquear el flujo del desarrollador. El CI es completo (tipos + seguridad + tests) para garantizar calidad antes del merge. El Cap 10 implementa esto con pre-commit y GitHub Actions.
