Módulo 1 — Por qué testear · Capítulo 1

Fundamentos y vocabulario

Unit · Integration · E2E · caja negra · caja blanca · stub · mock · spy · fake

Por qué existe este vocabulario — y por qué importa

Antes de escribir un solo test hay que hablar el mismo idioma. El vocabulario de testing no es burocracia: define qué estás probando, con qué nivel de aislamiento y qué tipo de evidencia te da sobre el comportamiento del sistema.

Un equipo que confunde "unit test" con "test que llama a la base de datos" tiene dos problemas: el primero es técnico, el segundo es de comunicación. Ambos generan bugs que llegan a producción.

INFO Este capítulo es deliberadamente conceptual. No hay herramientas todavía. El objetivo es que cuando llegues al Cap 5 (pytest) y Cap 9 (mocks) sepas exactamente qué estás configurando y por qué, en lugar de copiar comandos sin entenderlos.

La pirámide de testing — niveles y su razón de ser

La pirámide de testing describe cuántos tests de cada tipo deberías tener y por qué. No es una regla sagrada, pero sí una heurística útil que ha sobrevivido décadas de industria.

              /▲\
             / E2E \         ← pocos, lentos, frágiles, mucho valor de confianza
            /───────\
           /Integration\     ← moderados, dependen de sistema externo real o simulado
          /─────────────\
         /  Unit Tests   \   ← muchos, rápidos, aislados, feedback en milisegundos
        /─────────────────\

Velocidad de feedback:   Unit < 1ms | Integration ~100ms-5s | E2E segundos-minutos
Coste de mantenimiento:  Unit bajo  | Integration medio      | E2E alto
Aislamiento:             Unit total | Integration parcial     | E2E ninguno

Tests unitarios

Prueban una sola unidad de comportamiento en completo aislamiento. "Unidad" no significa "función" necesariamente — significa el comportamiento más pequeño que tiene sentido verificar por sí solo. No hablan con bases de datos, no hacen llamadas de red, no tocan el sistema de ficheros. Todo lo externo está simulado.

Tests de integración

Prueban que dos o más componentes funcionan correctamente juntos. Pueden involucrar una base de datos real (o de test), un broker de mensajes, una API externa. Son más lentos y más frágiles que los unitarios, pero validan contratos entre componentes.

Tests End-to-End (E2E)

Prueban el flujo completo desde la perspectiva del usuario o del sistema cliente. En una API REST: lanzar una petición HTTP real y verificar la respuesta completa. En una webapp: navegar con un navegador real. Son los más caros de mantener pero los que más confianza dan sobre el sistema completo.

NivelQué pruebaVelocidadCuántos tenerCuándo fallan
Unit Lógica de negocio pura, algoritmos, transformaciones < 1 ms ~70% del total Regressions en lógica interna
Integration Contratos entre capas, ORM, repositorios, adaptadores 100 ms — 5 s ~20% del total Incompatibilidades entre componentes
E2E Flujos críticos de negocio completos segundos ~10% del total Bugs de integración end-to-end, regresiones de UI
WARN Un antipatrón habitual: invertir la pirámide. Muchos E2E, pocos unitarios. El resultado es una suite que tarda 20 minutos en ejecutarse, falla aleatoriamente por timeouts de red o estado de la base de datos, y da feedback tan tarde que el desarrollador ya cambió de contexto. El ciclo de feedback lento destruye la disciplina de testing.

Caja negra vs caja blanca

Son dos perspectivas para diseñar qué casos de prueba escribir. No son excluyentes — un buen suite usa ambas en distintas proporciones según el contexto.

Caja negra

El tester no conoce ni le importa la implementación interna. Solo se fija en las entradas y las salidas esperadas. Prueba el contrato de la función/módulo. Si la implementación cambia pero el contrato se mantiene, el test no cambia.

Caja blanca

El tester conoce la implementación y diseña los casos para cubrir cada rama, cada condición, cada camino del código. Maximiza la cobertura de líneas y ramas. Íntimamente ligada a la complejidad ciclomática (Cap 2).

perspectiva_caja_negra.py — testear el contrato, no la implementación 
def calcular_descuento(precio: float, nivel_cliente: str) -> float:
    """
    Aplica descuento según nivel: 'vip' 20%, 'premium' 10%, resto 0%.
    """
    if nivel_cliente == "vip":
        return precio * 0.80
    elif nivel_cliente == "premium":
        return precio * 0.90
    return precio


# CAJA NEGRA: los tests no saben que hay un if/elif dentro.
# Solo verifican la promesa de la función.
def test_descuento_vip():
    assert calcular_descuento(100.0, "vip") == 80.0

def test_descuento_premium():
    assert calcular_descuento(100.0, "premium") == 90.0

def test_cliente_sin_descuento():
    assert calcular_descuento(100.0, "basico") == 100.0

def test_precio_cero_vip():
    assert calcular_descuento(0.0, "vip") == 0.0
perspectiva_caja_blanca.py — diseñar casos para cubrir cada rama 
# La misma función, pero esta vez diseñamos los tests
# mirando el código fuente para cubrir cada rama:
#
# Rama 1: nivel_cliente == "vip"     → precio * 0.80
# Rama 2: nivel_cliente == "premium" → precio * 0.90
# Rama 3: else                        → precio sin cambio
# Caso borde: precio negativo (¿está permitido?)
# Caso borde: nivel_cliente vacío ""
# Caso borde: nivel_cliente en mayúsculas "VIP"

def test_rama_vip():
    assert calcular_descuento(200.0, "vip") == 160.0

def test_rama_premium():
    assert calcular_descuento(200.0, "premium") == 180.0

def test_rama_else():
    assert calcular_descuento(200.0, "normal") == 200.0

def test_nivel_mayusculas_no_reconocido():
    # "VIP" != "vip" → caerá en else, precio sin descuento
    # Puede ser un bug o un comportamiento esperado.
    # Documentarlo con el test es valioso.
    assert calcular_descuento(100.0, "VIP") == 100.0

def test_precio_negativo():
    # La función no valida. ¿Debería? El test revela el gap.
    assert calcular_descuento(-50.0, "vip") == -40.0
INFO En la práctica, los tests de caja negra son los más mantenibles: si refactorizas la implementación sin cambiar el contrato, no tienes que tocar los tests. La caja blanca es útil para encontrar ramas no cubiertas y casos borde que el diseñador del contrato no documentó. Usa caja negra como base, caja blanca para completar.

Vocabulario de dobles de test — Mock, Stub, Spy, Fake

Este es el vocabulario que más confusión genera. La terminología viene de Gerard Meszaros (xUnit Test Patterns, 2007) y es la que usa la industria. En Python, la librería unittest.mock los unifica bajo el nombre "mock", lo que añade más confusión.

Stub

Devuelve respuestas predefinidas. No verifica nada. Su trabajo es proporcionar datos a la unidad bajo test para que pueda ejecutar su lógica. Ejemplo: una clase que siempre devuelve True en usuario_existe().

Mock

Un objeto que verifica que fue llamado correctamente. Tiene expectativas: "este método debe ser llamado con estos argumentos, N veces". Si las expectativas no se cumplen, el test falla. Su trabajo es verificar comportamiento.

Spy

Como el Mock pero no bloquea la implementación real. Deja que el código real se ejecute y además registra las llamadas para que puedas verificarlas después. Útil cuando necesitas el comportamiento real pero también quieres observarlo.

Fake

Una implementación funcional simplificada. No es un objeto vacío — tiene lógica real, pero más sencilla. Ejemplo: una base de datos en memoria en lugar de PostgreSQL. Los Fakes se usan en tests de integración, no en unitarios.

dobles_de_test.py — los cuatro tipos en código Python 
from unittest.mock import MagicMock, patch
from typing import Protocol


class RepositorioUsuarios(Protocol):
    def existe(self, user_id: int) -> bool: ...
    def guardar(self, user_id: int, datos: dict) -> None: ...


class ServicioRegistro:
    def __init__(self, repo: RepositorioUsuarios) -> None:
        self._repo = repo

    def registrar(self, user_id: int, datos: dict) -> str:
        if self._repo.existe(user_id):
            return "ya_existe"
        self._repo.guardar(user_id, datos)
        return "creado"


# ── STUB: proporciona dato fijo, no verifica llamadas ─────────
class StubRepositorioVacio:
    def existe(self, user_id: int) -> bool:
        return False  # siempre dice que no existe

    def guardar(self, user_id: int, datos: dict) -> None:
        pass  # no hace nada con los datos

def test_registro_usuario_nuevo_con_stub():
    stub = StubRepositorioVacio()
    servicio = ServicioRegistro(stub)
    resultado = servicio.registrar(1, {"nombre": "Ana"})
    assert resultado == "creado"


# ── MOCK: verifica que se llamó con los argumentos correctos ─
def test_guardar_es_llamado_con_datos_correctos():
    mock_repo = MagicMock(spec=RepositorioUsuarios)
    mock_repo.existe.return_value = False  # stub behavior
    servicio = ServicioRegistro(mock_repo)

    servicio.registrar(42, {"nombre": "Ana"})

    # Verificación de comportamiento: ¿se llamó guardar con los datos correctos?
    mock_repo.guardar.assert_called_once_with(42, {"nombre": "Ana"})


# ── FAKE: implementación real simplificada en memoria ─────────
class FakeRepositorio:
    def __init__(self):
        self._datos: dict[int, dict] = {}

    def existe(self, user_id: int) -> bool:
        return user_id in self._datos

    def guardar(self, user_id: int, datos: dict) -> None:
        self._datos[user_id] = datos

def test_no_permite_registro_duplicado_con_fake():
    fake = FakeRepositorio()
    servicio = ServicioRegistro(fake)

    servicio.registrar(1, {"nombre": "Ana"})
    resultado = servicio.registrar(1, {"nombre": "Ana duplicada"})

    assert resultado == "ya_existe"
    assert fake._datos[1]["nombre"] == "Ana"  # el original no fue sobreescrito
FAIL Confusión habitual: usar un Mock cuando necesitas un Stub, y vice versa. Si usas un Mock para proporcionar datos (rol de Stub), acabas con tests que verifican detalles de implementación que no importan (cuántas veces se llamó existe()). Resultado: tests frágiles que se rompen con refactors que no cambian el comportamiento. Regla: Stub para estado, Mock para comportamiento.
SUCCESS Resumen ejecutivo del vocabulario: Stub = "te doy el dato que necesitas". Mock = "quiero comprobar que me llamaste bien". Spy = "te dejo hacer lo tuyo pero tomo nota". Fake = "soy una versión simplificada que funciona de verdad". En Python, MagicMock puede hacer los cuatro roles — el peligro es no saber cuál estás usando.
