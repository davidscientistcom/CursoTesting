Hypothesis — Property-Based Testing

@given · strategies · shrinking · propiedades vs ejemplos · cuándo usar Hypothesis

El problema de los tests basados en ejemplos

Los tests de ejemplo (assert suma(2, 3) == 5) tienen un punto ciego: solo verifican los casos que el programador pensó. La realidad es que los bugs viven en los casos que no pensaste. Hypothesis los encuentra.

Tests de ejemplo           Tests basados en propiedades (Hypothesis)
───────────────────────    ─────────────────────────────────────────────
Tú eliges los inputs       Hypothesis genera cientos de inputs aleatorios
                           con sesgo hacia casos borde

Verificas: "para x=2       Verificas: "para CUALQUIER x e y, la propiedad
y y=3, el resultado es 5"  P(x,y) siempre se cumple"

Cobertura de casos:        Cobertura de casos:
  limitada a lo que           muy alta — incluye None, 0, -1, "", [], inf,
  el programador imagina      strings unicode, listas vacías, valores extremos

Cuando falla, sabes         Cuando falla, Hypothesis hace SHRINKING:
exactamente qué         →   reduce el input hasta el ejemplo MÍNIMO
                            que reproduce el fallo. Luego lo guarda
                            en base de datos para reproducir en futuros runs.

@given y strategies — los fundamentos

hypothesis_basico.py — primer test con Hypothesis 
pip install hypothesis
hypothesis_primer_test.py — comparando ejemplo-based vs property-based 
from hypothesis import given, assume
from hypothesis import strategies as st


def invertir_lista(lista: list) -> list:
    return lista[::-1]


# ── Test de ejemplo (tradicional) ───────────────────────────────
def test_invertir_lista_ejemplo():
    assert invertir_lista([1, 2, 3]) == [3, 2, 1]
    assert invertir_lista([]) == []


# ── Test de propiedad (Hypothesis) ──────────────────────────────
# Propiedad: invertir dos veces devuelve la lista original
@given(st.lists(st.integers()))
def test_doble_inversion_es_identidad(lista):
    assert invertir_lista(invertir_lista(lista)) == lista


# Hypothesis ejecuta este test con ~100 ejemplos generados automáticamente.
# Incluirá: [], [0], [-1], [2**63], [0, 0, 0], [1, -9999, 0], etc.


# Propiedad: la longitud se preserva
@given(st.lists(st.integers()))
def test_inversion_preserva_longitud(lista):
    assert len(invertir_lista(lista)) == len(lista)


# Propiedad: los elementos son los mismos (el orden cambia)
@given(st.lists(st.integers()))
def test_inversion_preserva_elementos(lista):
    assert sorted(invertir_lista(lista)) == sorted(lista)
hypothesis_strategies.py — catálogo de strategies más usadas 
from hypothesis import given
from hypothesis import strategies as st


# ── Tipos básicos ────────────────────────────────────────────────
@given(st.integers())                         # cualquier int, incluye extremos
@given(st.integers(min_value=0, max_value=100))  # rango acotado
@given(st.floats(allow_nan=False, allow_infinity=False))
@given(st.text())                             # cualquier string unicode
@given(st.text(alphabet=st.characters(whitelist_categories=("Lu", "Ll"))))
@given(st.booleans())
@given(st.none())


# ── Colecciones ──────────────────────────────────────────────────
@given(st.lists(st.integers()))               # listas de enteros
@given(st.lists(st.integers(), min_size=1))   # al menos un elemento
@given(st.lists(st.integers(), max_size=10))  # máximo 10 elementos
@given(st.sets(st.integers()))
@given(st.dictionaries(st.text(), st.integers()))
@given(st.tuples(st.integers(), st.text()))   # tupla (int, str)


# ── Combinadores ─────────────────────────────────────────────────
@given(st.one_of(st.integers(), st.text(), st.none()))  # any of
@given(st.integers() | st.text())                       # equivalente
@given(st.from_regex(r"\d{4}-\d{2}-\d{2}"))            # fechas ISO


# ── Ejemplo completo: función de parseo ─────────────────────────
def parsear_entero_seguro(valor: str) -> int | None:
    try:
        return int(valor)
    except ValueError:
        return None


@given(st.integers())
def test_parsear_entero_desde_su_representacion_string(n):
    """Si conviertes un int a str y lo parseas, debes recuperar el int original."""
    assert parsear_entero_seguro(str(n)) == n


@given(st.text())
def test_parsear_string_arbitrario_nunca_lanza_excepcion(texto):
    """La función no debe propagar excepciones nunca."""
    resultado = parsear_entero_seguro(texto)
    assert resultado is None or isinstance(resultado, int)

Shrinking — el superpoder de Hypothesis

Cuando Hypothesis encuentra un fallo, no te muestra el input de 500 elementos que falló — reduce automáticamente el input hasta el ejemplo más pequeño que reproduce el mismo fallo. Esto se llama shrinking y hace que los fallos sean inmediatamente comprensibles.

hypothesis_shrinking.py — ver shrinking en acción 
from hypothesis import given
from hypothesis import strategies as st


def ordenar_y_deduplicar(lista: list[int]) -> list[int]:
    """Ordena y elimina duplicados. Tiene un bug: no maneja negativos."""
    if any(x < 0 for x in lista):
        raise ValueError("Negativos no soportados")  # BUG: restricción no documentada
    return sorted(set(lista))


@given(st.lists(st.integers()))
def test_resultado_esta_ordenado(lista):
    """Propiedad: el resultado debe estar siempre ordenado."""
    resultado = ordenar_y_deduplicar(lista)
    assert resultado == sorted(resultado)


# Cuando ejecutas este test, Hypothesis:
# 1. Genera listas aleatorias hasta encontrar una que hace fallar
#    (por ejemplo: [-394, 0, 17, -1, 23, 45, ...])
# 2. Shrinking: reduce la lista → [-394, 0] → [-1] → [-1]
# 3. Te reporta: "Falsifying example: lista=[-1]"
#
# Hypothesis 8.x output:
# ─────────────────────────────────────────────────
# Falsifying example: test_resultado_esta_ordenado(
#   lista=[-1],
# )
# ValueError: Negativos no soportados
# ─────────────────────────────────────────────────
#
# El ejemplo mínimo que reproduce el fallo es [-1], no la lista original.
# Hypothesis guarda este caso en .hypothesis/examples para reproducirlo siempre.


# ── La base de datos de Hypothesis ─────────────────────────────
# .hypothesis/
#   examples/       ← casos que fallaron, siempre se re-ejecutan primero
#
# Comportamiento: si arreglas el bug, en el próximo run Hypothesis
# verifica el caso guardado primero. Luego explora nuevos casos.
# Nunca "olvida" un fallo que ya encontró.

Propiedades útiles — cómo pensar en propiedades

La parte difícil de Hypothesis no es la sintaxis — es saber qué propiedad verificar. Aquí están los patrones más comunes:

hypothesis_patrones.py — patrones de propiedades para casos reales 
from hypothesis import given, assume
from hypothesis import strategies as st
from decimal import Decimal


# ── Patrón 1: Round-trip (serializar/deserializar) ──────────────
import json

@given(st.dictionaries(
    st.text(min_size=1),
    st.one_of(st.integers(), st.text(), st.booleans(), st.none())
))
def test_json_roundtrip(datos):
    """Serializar a JSON y deserializar debe dar el mismo objeto."""
    assert json.loads(json.dumps(datos)) == datos


# ── Patrón 2: Idempotencia ──────────────────────────────────────
@given(st.text())
def test_strip_es_idempotente(texto):
    """Aplicar strip dos veces es lo mismo que una."""
    assert texto.strip().strip() == texto.strip()


# ── Patrón 3: Invariantes de dominio ──────────────────────────────
def aplicar_descuento(precio: Decimal, porcentaje: int) -> Decimal:
    assert 0 <= porcentaje <= 100
    return precio * Decimal(100 - porcentaje) / Decimal(100)

@given(
    precio=st.decimals(min_value="0.01", max_value="9999.99", places=2),
    porcentaje=st.integers(min_value=0, max_value=100),
)
def test_descuento_nunca_aumenta_el_precio(precio, porcentaje):
    """Un descuento nunca puede hacer el precio mayor."""
    resultado = aplicar_descuento(precio, porcentaje)
    assert resultado <= precio

@given(
    precio=st.decimals(min_value="0.01", max_value="9999.99", places=2),
    porcentaje=st.integers(min_value=0, max_value=100),
)
def test_descuento_nunca_produce_precio_negativo(precio, porcentaje):
    resultado = aplicar_descuento(precio, porcentaje)
    assert resultado >= Decimal("0")


# ── Patrón 4: Comparación con implementación simple ─────────────
def suma_rapida(numeros: list[int]) -> int:
    """Implementación optimizada (podría tener bugs)."""
    if not numeros:
        return 0
    return numeros[0] + suma_rapida(numeros[1:])

@given(st.lists(st.integers(min_value=-1000, max_value=1000), max_size=50))
def test_suma_rapida_equivale_a_builtin(numeros):
    """La implementación custom debe coincidir con sum() de Python."""
    assert suma_rapida(numeros) == sum(numeros)


# ── Patrón 5: assume() — descartar inputs inválidos ─────────────
@given(st.integers(), st.integers())
def test_division_entera(a, b):
    assume(b != 0)  # descarta el caso b=0 (no es un bug, es precondición)
    resultado = a // b
    assert isinstance(resultado, int)
    # No hay assertion sobre el valor — solo que no lanza excepción
    # y que el resultado es del tipo correcto
INFO assume(condición) descarta el input si la condición es falsa y genera uno nuevo. Úsalo con moderación: si assume descarta el 99% de los inputs, Hypothesis lo detecta y falla con UnsatisfiedAssumption. En ese caso, es mejor usar una strategy acotada (st.integers(min_value=1) en lugar de assume(n != 0)).

Hypothesis con dataclasses y objetos de dominio

hypothesis_dataclasses.py — generar objetos de dominio automáticamente 
from hypothesis import given
from hypothesis import strategies as st
from hypothesis.strategies import builds
from dataclasses import dataclass
from decimal import Decimal


@dataclass
class LineaPedido:
    sku: str
    cantidad: int
    precio_unitario: Decimal


@dataclass
class Pedido:
    lineas: list[LineaPedido]

    def total(self) -> Decimal:
        return sum(l.precio_unitario * l.cantidad for l in self.lineas)

    def cantidad_total(self) -> int:
        return sum(l.cantidad for l in self.lineas)


# Strategy para generar LineaPedido válidas
linea_strategy = builds(
    LineaPedido,
    sku=st.text(min_size=3, max_size=10, alphabet="ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789"),
    cantidad=st.integers(min_value=1, max_value=100),
    precio_unitario=st.decimals(min_value="0.01", max_value="999.99", places=2),
)

pedido_strategy = builds(
    Pedido,
    lineas=st.lists(linea_strategy, min_size=1, max_size=20),
)


@given(pedido_strategy)
def test_total_pedido_nunca_es_negativo(pedido: Pedido):
    assert pedido.total() >= Decimal("0")


@given(pedido_strategy)
def test_cantidad_total_coincide_con_suma_de_lineas(pedido: Pedido):
    esperado = sum(l.cantidad for l in pedido.lineas)
    assert pedido.cantidad_total() == esperado


@given(pedido_strategy, pedido_strategy)
def test_combinar_pedidos_suma_los_totales(pedido_a: Pedido, pedido_b: Pedido):
    pedido_combinado = Pedido(lineas=pedido_a.lineas + pedido_b.lineas)
    assert pedido_combinado.total() == pedido_a.total() + pedido_b.total()

Cuándo usar Hypothesis y cuándo no

ContextoHypothesisTests de ejemplo
Funciones matemáticas (suma, descuento, conversión) Ideal — las propiedades matemáticas son evidentes Complementario para casos específicos documentados
Parsers / serialización (JSON, CSV, protobuf) Ideal — round-trip property es siempre aplicable Para formatos de entrada específicos
Algoritmos de ordenación, búsqueda Ideal — comparar con implementación de referencia Para casos borde explícitos (lista vacía, un elemento)
Lógica de negocio con reglas de dominio Muy útil — invariantes de dominio como propiedades Para casos de negocio específicos con valores fijos
UI / HTTP handlers / efectos secundarios Difícil — las propiedades son menos evidentes Más adecuado — comportamiento concreto esperado
Flujos de integración complejos No recomendado — demasiado acoplado a infraestructura Tests de integración con escenarios fijos
SUCCESS Regla práctica: añade Hypothesis a una función cuando, al escribir @pytest.mark.parametrize, notes que estás generando muchos casos similares manualmente. Si tienes 15 pares de entrada/salida en parametrize y todos siguen el mismo patrón, Hypothesis probablemente encontrará el fallo que te faltó.
