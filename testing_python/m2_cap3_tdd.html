Módulo 2 — TDD · Capítulo 3

TDD: el test dirige el diseño

Red-Green-Refactor · diseño emergente · TDD vs test-after · sin mocks no hay TDD real

Qué es TDD y qué no es

TDD (Test-Driven Development) no es "escribir tests antes del código". Eso es una consecuencia. TDD es usar los tests como herramienta de diseño: el test te fuerza a pensar en la interfaz, el contrato y los casos borde antes de pensar en la implementación.

El resultado secundario es que tienes tests. El resultado principal es que tienes mejor diseño.

Fase 1 — Red

Escribe un test que falla

El test describe el comportamiento que quieres. No existe implementación. El test debe fallar por la razón correcta, no por error de sintaxis.

Fase 2 — Green

Haz el test pasar

Escribe el mínimo código necesario para que el test pase. No más. No te anticipes a requisitos futuros. La implementación puede ser fea.

Fase 3 — Refactor

Limpia sin romper

Mejora el diseño del código con los tests como red de seguridad. Si algo se rompe, lo sabes en segundos. Vuelve al estado anterior y refactoriza más pequeño.

INFO El ciclo Red-Green-Refactor no es de horas sino de minutos. Un ciclo típico tarda 2-5 minutos. Hacer TDD de forma disciplinada significa hacer decenas de ciclos en una jornada. Si tu ciclo tarda 30 minutos, algo está mal: o el test es demasiado grande, o la implementación más mínima que buscas no es mínima.

TDD en la práctica — ciclo completo paso a paso

Vamos a implementar un servicio de autenticación de usuarios con TDD. No existe ningún código de producción todavía. El test va primero.

paso_1_RED.py — test que describe el comportamiento deseado. Falla porque no existe la clase 
# test_auth.py — escribimos esto PRIMERO, antes de ningún código de producción

from auth import ServicioAuth, CredencialesInvalidas  # no existen todavía


def test_login_exitoso_devuelve_token():
    """Un usuario con credenciales válidas obtiene un token."""
    servicio = ServicioAuth()
    token = servicio.login("ana@ejemplo.com", "password123")
    assert token is not None
    assert len(token) > 0


# $ pytest test_auth.py
# ERROR: ModuleNotFoundError: No module named 'auth'
#
# RED: el test falla. Razón correcta: no existe la implementación.
# Ahora sabemos qué interfaz queremos: ServicioAuth(), .login(email, password).
paso_2_GREEN.py — implementación mínima para pasar el test. Sin sobreingeniería 
# auth.py — el mínimo para hacer pasar el test

import secrets


class CredencialesInvalidas(Exception):
    pass


class ServicioAuth:
    def login(self, email: str, password: str) -> str:
        # Implementación mínima: cualquier credencial genera un token
        return secrets.token_hex(32)


# $ pytest test_auth.py
# PASSED ✓
#
# GREEN: el test pasa. La implementación es fea (no valida nada),
# pero es el mínimo para satisfacer el contrato definido por el test.
# No escribimos más de lo necesario.
paso_3_NEXT_RED.py — añadir el siguiente comportamiento: credenciales inválidas 
# test_auth.py — añadimos el segundo comportamiento

from auth import ServicioAuth, CredencialesInvalidas
import pytest


def test_login_exitoso_devuelve_token():
    servicio = ServicioAuth()
    token = servicio.login("ana@ejemplo.com", "password123")
    assert token is not None
    assert len(token) > 0


def test_login_fallido_lanza_excepcion():
    """Credenciales incorrectas deben lanzar CredencialesInvalidas."""
    servicio = ServicioAuth()
    with pytest.raises(CredencialesInvalidas):
        servicio.login("fake@ejemplo.com", "wrong")


# $ pytest test_auth.py
# .F
# FAILED test_auth.py::test_login_fallido_lanza_excepcion
# ServicioAuth.login no lanza la excepción → RED.
paso_4_GREEN_REFACTOR.py — implementación real + refactor con mock del repositorio 
# auth.py — ahora la implementación real emerge del test
# Nota: el test nos fuerza a pensar en cómo validamos credenciales.
# Resultado: necesitamos un repositorio de usuarios. TDD nos da la arquitectura.

import secrets
from typing import Protocol


class RepositorioUsuarios(Protocol):
    def verificar_credenciales(self, email: str, password: str) -> bool: ...


class CredencialesInvalidas(Exception):
    pass


class ServicioAuth:
    def __init__(self, repo: RepositorioUsuarios | None = None) -> None:
        # Si no se inyecta repo, usamos uno en memoria para desarrollo
        self._repo = repo or _RepoEnMemoria()

    def login(self, email: str, password: str) -> str:
        if not self._repo.verificar_credenciales(email, password):
            raise CredencialesInvalidas(f"Credenciales inválidas para {email}")
        return secrets.token_hex(32)


class _RepoEnMemoria:
    """Implementación por defecto para desarrollo. En producción se inyecta el real."""
    _usuarios = {"ana@ejemplo.com": "password123"}

    def verificar_credenciales(self, email: str, password: str) -> bool:
        return self._usuarios.get(email) == password


# $ pytest test_auth.py
# ..
# PASSED ✓ GREEN con diseño real.
# El TDD nos forzó a:
# 1. Definir la interfaz (login(email, password) -> str)
# 2. Separar Auth de la fuente de datos (RepositorioUsuarios Protocol)
# 3. Hacer el código testeable por diseño (inyección en __init__)

TDD vs test-after — por qué el orden cambia el diseño

Escribir tests antes y después del código produce resultados radicalmente distintos, aunque al final ambos tengan tests.

TDD (test primero)Test-after (test después)
Diseño de la interfaz Lo define el test antes de implementar. Siempre limpio. Lo define la implementación. El test se adapta a lo que hay.
Testabilidad Garantizada: si algo no es testeable, no puedes avanzar. Puede ser muy baja si la implementación es acoplada.
Cobertura de casos borde Alta: piensas en los casos antes de implementar. Baja: tiendes a testear el camino feliz que ya escribiste.
Over-engineering Bajo: solo implementas lo que el test requiere. Alto: implementas "lo que crees que hará falta" sin test que lo guíe.
Confianza al refactorizar Alta: los tests son la red de seguridad del refactor. Media o baja: los tests se escribieron para el código actual.
WARN El mayor error al aprender TDD es aplicarlo religiosamente a todo desde el primer día. TDD tiene coste de aprendizaje. Aplícalo primero a la lógica de negocio pura (donde más valor da), luego extiéndelo. No lo apliques a código de configuración, scripts de migración o pruebas de concepto.

Sin mocks no hay TDD real — el eslabón que une todo

Aquí está el punto que más se omite en los tutoriales de TDD: si no sabes usar mocks, no puedes hacer TDD a nivel unitario.

Cuando el test del Cap 3 llegó a "necesito verificar credenciales", tuvimos que decidir: ¿cómo isolo el ServicioAuth de la base de datos real? La respuesta es el Protocol + mock en el test. Si no sabemos mockear, o bien:

  • Levantamos una base de datos real en cada test (lento, frágil — deja de ser unitario)
  • No podemos separar ServicioAuth del repositorio (alto acoplamiento — el ciclo Cap 2)
  • Abandonamos TDD porque "es muy difícil" (en realidad, es que el código no está diseñado para ello)
tdd_con_mock.py — test real de ServicioAuth usando mock del repositorio 
from unittest.mock import MagicMock
from auth import ServicioAuth, CredencialesInvalidas
import pytest


# El test no necesita ninguna base de datos.
# El mock simula el RepositorioUsuarios con el comportamiento que queremos.

def test_login_llama_al_repo_con_credenciales_correctas():
    """TDD: verificar que ServicioAuth delega la verificación al repositorio."""
    mock_repo = MagicMock()
    mock_repo.verificar_credenciales.return_value = True

    servicio = ServicioAuth(repo=mock_repo)
    servicio.login("ana@ejemplo.com", "pass123")

    # El Mock verifica el COMPORTAMIENTO: ¿se llamó con los argumentos correctos?
    mock_repo.verificar_credenciales.assert_called_once_with(
        "ana@ejemplo.com", "pass123"
    )


def test_login_lanza_excepcion_si_repo_dice_falso():
    """Si repo dice que las credenciales son incorrectas, debe lanzar la excepción."""
    mock_repo = MagicMock()
    mock_repo.verificar_credenciales.return_value = False  # Stub behavior

    servicio = ServicioAuth(repo=mock_repo)

    with pytest.raises(CredencialesInvalidas):
        servicio.login("hacker@ejemplo.com", "wrong")


# Sin el mock, estos tests necesitarían:
# - Una base de datos corriendo
# - Datos de prueba cargados
# - Limpieza después del test
# → El mock hace que el test sea instantáneo, determinista y aislado.

No son capítulos aislados. Se retroalimentan.
SUCCESS TDD no es una religión. Es una herramienta de diseño con una propiedad muy valiosa: si no puedes escribir el test primero, es una señal de que el diseño tiene problemas. Un código que solo es testeable con toda la infraestructura levantada no está bien diseñado. El dolor de hacer TDD en código acoplado es el detector de problemas de arquitectura más honesto que existe.
