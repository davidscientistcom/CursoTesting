Módulo 7 — Automatización · Capítulo 10

Git hooks y automatización

pre-commit · hooks nativos de git · pipeline ruff + mypy + pytest + coverage · GitHub Actions

Por qué automatizar — el problema del "lo haré manualmente"

La calidad manual no escala. Cada vez que alguien olvida ejecutar el linter, o hace un commit sin pasar los tests, la deuda técnica crece silenciosamente. Los git hooks convierten las comprobaciones de calidad en inevitables: no puedes hacer commit si algo falla.

Sin automatización:          Con git hooks + CI:
────────────────────         ──────────────────────────────────────
Developer escribe código     Developer escribe código
      ↓                            ↓
git commit (sin checks)      git commit
      ↓                            ↓
push al repo                 pre-commit hook ejecuta (local, <10s):
      ↓                            ruff format + ruff check
CI falla 5 minutos después         mypy (rápido)
      ↓                            pytest -m "not slow"
Developer vuelve al                ↓ (si todo pasa → commit creado)
contexto anterior           push al repo
      ↓                            ↓
Fix rápido que introduce     CI completo ejecuta (<4min):
otro bug                           pytest completo + coverage
      ↓                            bandit
Ciclo de deuda técnica             ↓ (si todo pasa → merge habilitado)
                             Deploy seguro

Git hooks nativos — el enfoque manual

Git ejecuta scripts en .git/hooks/ en respuesta a eventos. No se versionan con el repositorio (por eso pre-commit framework es mejor para equipos).

.git/hooks/pre-commit — hook nativo (sin framework) 
#!/bin/bash
# .git/hooks/pre-commit
# Hacer ejecutable: chmod +x .git/hooks/pre-commit

set -e

echo "--- pre-commit: verificando calidad ---"

# 1. Formato
ruff format --check .
if [ $? -ne 0 ]; then
    echo "ERROR: código no formateado. Ejecuta: ruff format ."
    exit 1
fi

# 2. Linting
ruff check .
if [ $? -ne 0 ]; then
    echo "ERROR: problemas de linting. Ejecuta: ruff check --fix ."
    exit 1
fi

# 3. Tests rápidos (excluir tests lentos y de integración)
pytest -m "not slow and not integration" -q
if [ $? -ne 0 ]; then
    echo "ERROR: tests fallando."
    exit 1
fi

echo "--- pre-commit: OK ---"
exit 0
WARN Los hooks en .git/hooks/ no se versionan ni se comparten con el equipo. Cada developer tiene que instalarlos manualmente. Para equipos, usa pre-commit framework que sí se versiona en .pre-commit-config.yaml.

pre-commit framework — la forma correcta para equipos

pre-commit gestiona hooks como dependencias del proyecto: se configuran en .pre-commit-config.yaml, se instalan con un comando y se aplican a todos los miembros del equipo de forma idéntica.

instalacion_precommit.sh — setup inicial en un proyecto 
pip install pre-commit

# Instalar los hooks en el repositorio local
pre-commit install

# Ejecutar manualmente sobre todos los ficheros (primera vez o CI)
pre-commit run --all-files

# Ejecutar solo sobre los ficheros staged (es lo que hace el hook)
pre-commit run

# Actualizar todas las versiones de hooks a las últimas
pre-commit autoupdate
.pre-commit-config.yaml — configuración completa para Python 
repos:
  # ── Hooks básicos de git ─────────────────────────────────────────
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace           # elimina espacios al final de línea
      - id: end-of-file-fixer             # asegura newline al final del fichero
      - id: check-yaml                    # valida sintaxis YAML
      - id: check-toml                    # valida sintaxis TOML
      - id: check-merge-conflict          # detecta marcas de conflicto de merge
      - id: check-added-large-files       # bloquea ficheros > 1MB
        args: ["--maxkb=1024"]
      - id: no-commit-to-branch           # impide commits directos a main/master
        args: ["--branch", "main", "--branch", "master"]

  # ── ruff: linting + formato ──────────────────────────────────────
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.4
    hooks:
      - id: ruff
        args: [--fix, --exit-non-zero-on-fix]
      - id: ruff-format

  # ── mypy: type checking ──────────────────────────────────────────
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.10.0
    hooks:
      - id: mypy
        additional_dependencies: [types-requests, types-PyYAML]
        args: [--strict, --ignore-missing-imports]
        files: ^src/

  # ── bandit: seguridad (solo en src, no en tests) ─────────────────
  - repo: https://github.com/PyCQA/bandit
    rev: 1.7.9
    hooks:
      - id: bandit
        args: [-r, --skip, "B101", -ll]   # B101: assert (ok en tests)
        files: ^src/
hook_personalizado.py — añadir pytest al pre-commit hook 
# .pre-commit-config.yaml — añadir hook local de pytest
# (los hooks locales se definen en el mismo repo)

repos:
  # ... otros repos ...

  # ── Hook local: pytest rápido ────────────────────────────────────
  - repo: local
    hooks:
      - id: pytest-fast
        name: pytest (unit tests rápidos)
        entry: pytest
        language: system
        args:
          - -m
          - "not slow and not integration"
          - -q
          - --tb=short
        pass_filenames: false
        always_run: true

      - id: pytest-with-coverage
        name: pytest + coverage gate
        entry: pytest
        language: system
        args:
          - --cov=src
          - --cov-fail-under=80
          - --cov-branch
          - -q
          - --tb=short
        pass_filenames: false
        # Solo en commit a ramas de feature (no en cada cambio pequeño)
        stages: [push]

Pipeline completo con GitHub Actions

Los hooks locales son la primera línea de defensa. El CI es la segunda: se ejecuta en todos los pushes y Pull Requests, en un entorno limpio y reproducible.

.github/workflows/ci.yml — pipeline completo de CI 
name: CI

on:
  push:
    branches: ["**"]
  pull_request:
    branches: [main, develop]

jobs:
  quality:
    name: "Quality checks"
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: "pip"

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: Lint and format check (ruff)
        run: |
          ruff format --check src/ tests/
          ruff check src/ tests/

      - name: Type checking (mypy)
        run: mypy src/ --strict --ignore-missing-imports

      - name: Security scan (bandit)
        run: bandit -r src/ --exclude tests/ -ll

  tests:
    name: "Tests + Coverage"
    runs-on: ubuntu-latest
    needs: quality     # Solo ejecuta si quality pasa

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: "pip"

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: Run tests with coverage
        run: |
          pytest \
            --cov=src \
            --cov-report=xml \
            --cov-report=term-missing \
            --cov-branch \
            --cov-fail-under=80 \
            -v

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v4
        with:
          file: ./coverage.xml
          fail_ci_if_error: false

  integration:
    name: "Integration tests"
    runs-on: ubuntu-latest
    needs: tests

    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: "pip"

      - name: Install dependencies
        run: pip install -r requirements.txt -r requirements-dev.txt

      - name: Run integration tests
        env:
          DATABASE_URL: "postgresql://postgres:testpass@localhost:5432/testdb"
        run: pytest -m integration -v

Resumen: el pipeline completo de calidad

Herramienta          Cuándo             Qué verifica              Fallo bloquea
─────────────────────────────────────────────────────────────────────────────────
ruff format          pre-commit         Formato consistente        Commit local
ruff check           pre-commit         Linting, imports, bugs     Commit local
mypy                 pre-commit + CI    Tipos estáticos            Commit + PR
bandit               pre-commit + CI    Vulnerabilidades           Commit + PR
pytest (unit fast)   pre-commit         Tests rápidos sin infra    Commit local
pytest (completo)    CI                 Tests completos            PR merge
pytest-cov ≥80%      CI                 Gate de cobertura          PR merge

SUCCESS Una vez que el pipeline está en pie, la disciplina es gratuita: no necesitas acordarte de ejecutar nada. El equipo puede moverse rápido porque los hooks y el CI actúan como red de seguridad automática. La inversión inicial (2-4 horas de setup) se recupera en el primer bug que atrapa antes de llegar a producción.
