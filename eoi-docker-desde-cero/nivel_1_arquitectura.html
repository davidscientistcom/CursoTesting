01

Arquitectura y primer arranque

Antes de hablar de Dockerfiles o volumenes hay que fijar una idea basica: que piezas componen el laboratorio y que efecto produce cada comando inicial.

1. Por que aqui Docker si y host directo no

Este curso no parte de una opinion abstracta sobre contenedores. Parte de una necesidad docente muy concreta: poder destruir el laboratorio y reconstruirlo delante de la clase sin depender de configuraciones hechas a mano el dia anterior. Si instalases Jenkins, Gitea y el relay directamente en el host, una parte importante de la sesion se iria en explicar paquetes del sistema, servicios del init, rutas locales y pequenas diferencias del entorno anfitrion. Al contenedorizarlos, la topologia queda declarada y puede repetirse con el mismo guion.

Instalacion directa en el host
  • El estado queda repartido entre servicios del sistema y ficheros locales.
  • Es facil olvidar un cambio manual que no quedo documentado.
  • La reconstruccion completa es mas lenta y menos predecible.
Laboratorio con Docker Compose
  • Servicios, puertos, volumenes y variables quedan descritos en un mismo fichero.
  • docker compose down -v --remove-orphans borra el estado del lab de una forma controlada.
  • docker compose up -d --build recompone el stack con la configuracion declarada.
INFO
Idea clave: no se usa Docker porque sea moderno. Se usa porque reduce el numero de variables ocultas cuando tienes que explicar un sistema entero delante de otros.

2. Que servicios existen y que demuestra cada uno

El laboratorio real tiene tres servicios. Ese numero es lo bastante pequeno para seguirlo en directo, pero lo bastante rico para enseñar red interna, puertos publicados, persistencia y cooperacion entre contenedores.

GIT

Gitea

Es el repositorio Git local. Publica HTTP en 3000 y SSH en 222. Permite enseñar que un contenedor puede exponer servicios al host sin dejar de vivir dentro de una red privada.

JNK

Jenkins

Es el servicio mas interesante para Docker porque no solo arranca como contenedor: ademas necesita construir y lanzar otros contenedores durante el pipeline.

REL

Webhook relay

Es un servicio Python pequeno que recibe el evento de Gitea y llama a Jenkins desde la red interna del stack. Sirve para explicar DNS interno entre servicios.

3. Primer contacto con Compose: ver antes de arrancar

Antes de levantar nada, conviene pedirle a Compose una vista simplificada de la topologia. Esto reduce errores de lectura y da al alumno una lista concreta de lo que deberia aparecer despues.

bash - Ver los servicios declarados 
cd /home/deivit/Lucentia/projects/jenkins-lab
docker compose config --services

En la revalidacion de esta sesion, la salida fue exactamente esta:

text - Servicios observados 
gitea
jenkins
webhook-relay

Si aqui aparece un servicio que no esperabas, el problema esta en el YAML. Si aqui falta uno, el problema esta en el YAML. Todavia no has levantado nada, asi que este es el mejor momento para corregir el modelo antes de pasar a la ejecucion.

4. Arranque real paso a paso

El guion que hemos usado para revalidar el laboratorio no contiene magia. Son tres pasos enlazados: destruir, construir y observar. Lo importante es entender que cambia en el sistema despues de cada comando.

1
Limpiar el estado anterior

docker compose down -v --remove-orphans elimina contenedores, red y volumenes del laboratorio. Si haces esto, el siguiente arranque parte de cero. Si no lo haces, puedes estar observando restos de una ejecucion previa.

2
Reconstruir y arrancar

docker compose up -d --build obliga a revisar las imagenes y deja los contenedores corriendo en segundo plano. Si quitas --build, puedes seguir usando una imagen antigua. Si quitas -d, el terminal queda atado a los logs.

3
Comprobar el estado observable

docker compose ps te dice si el modelo que declaraste se corresponde con procesos reales en ejecucion.

bash - Secuencia minima reproducible 
cd /home/deivit/Lucentia/projects/jenkins-lab
docker compose down -v --remove-orphans
docker compose up -d --build
docker compose ps

Esta fue la salida relevante de docker compose ps durante la sesion:

text - Estado de contenedores observado 
NAME                  IMAGE                       SERVICE         STATUS                        PORTS
gitea-jenkins-relay   jenkins-lab-webhook-relay  webhook-relay   Up About a minute
gitea-lab             gitea/gitea:1.22.6         gitea           Up About a minute             3000->3000, 222->22
jenkins-lab           jenkins-lab-jenkins        jenkins         Up About a minute (healthy)   9090->8080, 50000->50000

Observa dos hechos importantes. Primero, el nombre de servicio y el nombre de contenedor no son lo mismo. Segundo, Jenkins aparece como healthy porque el Compose real define un healthcheck; eso se explicara mas adelante, pero aqui ya ves el efecto visible.

5. El recorte del compose que merece memorizar

No hace falta memorizar el fichero entero. Si estas empezando, conviene fijarse en las secciones que determinan el comportamiento: services, ports, volumes, environment y networks.

yaml - Fragmento real del docker-compose 
services:
  gitea:
    image: gitea/gitea:1.22.6
    ports:
      - "3000:3000"
      - "222:22"
    volumes:
      - gitea_data:/data

  jenkins:
    build:
      context: ./jenkins
    ports:
      - "9090:8080"
      - "50000:50000"
    volumes:
      - jenkins_data:/var/jenkins_home
      - /var/run/docker.sock:/var/run/docker.sock
      - /usr/bin/docker:/usr/bin/docker:ro

  webhook-relay:
    build:
      context: ./webhook-relay
    environment:
      JENKINS_URL: http://jenkins:8080

6. Dos fallos muy frecuentes al arrancar

WARN
Fallo tipico 1: Docker daemon caido. Si ejecutas docker compose up y Docker no responde, no estas ante un problema del YAML. Primero hay que recuperar el motor Docker.
FAIL
Fallo tipico 2: puerto ocupado en el host. Si otro proceso usa 3000 o 9090, el contenedor puede estar bien definido y aun asi no llegar a publicarse correctamente hacia fuera.

7. Que deberia quedarte claro al cerrar este capitulo

  • Docker aqui sirve para declarar y reconstruir una topologia completa, no solo para ejecutar un proceso aislado.
  • Compose te deja inspeccionar el modelo antes de arrancarlo y comprobar el resultado despues.
  • El laboratorio tiene tres servicios porque eso permite enseñar cooperacion real entre contenedores.
  • El siguiente paso natural es entender como se construyen las imagenes que despues Compose ejecuta.
