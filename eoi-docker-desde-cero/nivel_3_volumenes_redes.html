03

Redes, volumenes y mounts

En cuanto el laboratorio tiene mas de un contenedor, aparecen dos preguntas inevitables: como se hablan entre si y que partes del estado deben sobrevivir cuando destruyes el stack.

1. El sistema de archivos del contenedor no es tu disco duro

Un contenedor tiene su propio sistema de archivos en tiempo de ejecucion. Si escribes datos dentro y luego destruyes el contenedor, esos datos desaparecen salvo que los hayas sacado a un volumen o a un bind mount. Esta es una idea que conviene fijar muy pronto, porque cambia la forma de pensar sobre logs, bases de datos, configuracion y directorios de trabajo.

INFO
Si haces esto: borras un contenedor creado desde una imagen sin volumenes. Pasa esto: pierdes los datos escritos solo dentro de ese contenedor. Si haces lo otro: montas un volumen nombrado. Pasa esto otro: el contenedor puede desaparecer y el volumen sigue existiendo.

2. Ejemplo base: volumen nombrado frente a bind mount

Antes del laboratorio real, conviene distinguir dos mecanismos distintos. Los dos sirven para sacar algo fuera del sistema de archivos efimero del contenedor, pero no resuelven exactamente el mismo problema.

Volumen nombrado
  • Docker lo gestiona por ti.
  • Es apropiado para datos persistentes de servicios.
  • En el laboratorio se usa para gitea_data y jenkins_data.
Bind mount
  • Conecta una ruta del host con una ruta del contenedor.
  • Es apropiado cuando necesitas exponer una herramienta o un directorio real del host.
  • En el laboratorio se usa para /var/run/docker.sock y /usr/bin/docker.
yaml - Ejemplo base con los dos tipos 
services:
  app:
    image: python:3.12-slim
    volumes:
      - app_data:/var/app/data
      - ./src:/app/src:ro

volumes:
  app_data:

En ese ejemplo, app_data sirve para persistir datos propios de la aplicacion y ./src:/app/src:ro sirve para exponer codigo del host en modo solo lectura dentro del contenedor. El primer mount habla de persistencia. El segundo habla de visibilidad entre host y contenedor.

3. Caso real: lo que persiste en Jenkins Lab

El laboratorio usa volumenes para todo aquello que seria molesto perder en cada reinicio: los datos de Gitea y el directorio de trabajo de Jenkins. Si esos volumenes no existiesen, cada arranque seria una reinstalacion desde cero.

yaml - Volumenes y mounts reales del laboratorio 
services:
  gitea:
    volumes:
      - gitea_data:/data

  jenkins:
    volumes:
      - jenkins_data:/var/jenkins_home
      - /var/run/docker.sock:/var/run/docker.sock
      - /usr/bin/docker:/usr/bin/docker:ro

volumes:
  gitea_data:
  jenkins_data:

En esta sesion, Compose devolvio exactamente estos volumenes declarados:

text - Volumenes observados 
gitea_data
jenkins_data

4. Por que Jenkins monta el socket y el binario Docker

Esta es una de las partes mas utiles para clase porque rompe la idea ingenua de que un contenedor siempre vive completamente aislado. El Jenkins del laboratorio necesita hablar con el daemon Docker del host para lanzar otros contenedores durante el pipeline. Para eso no basta con tener instalado Jenkins: hace falta que dentro del contenedor exista acceso al socket del daemon y al binario docker.

01 Jenkins lee el pipeline
>
02 Docker sock expone el daemon
>
03 docker ejecuta comandos
>
04 Contenedores tests y base efimera
WARN
Consecuencia directa: si quitas el bind mount del socket o del binario Docker, Jenkins deja de poder lanzar contenedores del pipeline. El servicio puede arrancar, pero el flujo de CI se rompe en tiempo de ejecucion.

5. Redes: como se hablan entre si los servicios

En Docker Compose, los servicios que comparten red pueden encontrarse por nombre. Ese detalle ahorra mucha configuracion manual y hace que un contenedor pueda llamar a otro sin usar localhost del host. En el laboratorio, el relay no apunta a http://localhost:9090, sino a http://jenkins:8080. Esto solo funciona porque ambos servicios viven en la misma red de Compose.

yaml - Red declarada del laboratorio 
services:
  gitea:
    networks:
      - cicd_net

  jenkins:
    networks:
      - cicd_net

  webhook-relay:
    networks:
      - cicd_net

networks:
  cicd_net:
    driver: bridge

Esa definicion significa que los tres servicios comparten una red bridge llamada cicd_net. Desde dentro de esa red, jenkins es un nombre resoluble, igual que gitea o webhook-relay.

6. Verificacion observable de red y estado

No basta con confiar en el YAML. Conviene comprobar que la topologia se ha materializado. Durante la sesion, al listar redes del host aparecio la red declarada por el proyecto:

text - Red visible en el host 
jenkins-lab_cicd_net

Si esta red no aparece, Compose no ha levantado el proyecto como esperabas. Si aparece la red pero un servicio no puede alcanzar a otro por nombre, el problema suele estar en el nombre de servicio, en la variable de entorno usada o en que ambos no comparten realmente la misma red.

7. Tres fallos que merece la pena explicar en clase

FAIL
Fallo 1: usar localhost dentro de un contenedor para referirse a otro servicio. Dentro del contenedor, localhost es ese contenedor, no el servicio vecino.
FAIL
Fallo 2: destruir el stack con down -v y esperar que Jenkins o Gitea conserven su estado. Al borrar volumenes, estas pidiendo precisamente lo contrario.
WARN
Fallo 3: confundir un volumen nombrado con una carpeta del proyecto. Un volumen Docker no vive como un directorio legible al lado del docker-compose.yml; lo gestiona Docker en su almacenamiento interno.

8. Cierre del capitulo

  • La persistencia no sucede por accidente: hay que declararla.
  • Redes y volumenes resuelven problemas distintos y conviene explicarlos por separado.
  • El Jenkins del laboratorio demuestra un caso real de bind mount al host para poder usar Docker desde dentro del contenedor.
  • El siguiente paso es volver al fichero Compose completo para leerlo como una declaracion operativa y no como una lista de claves YAML sin contexto.
