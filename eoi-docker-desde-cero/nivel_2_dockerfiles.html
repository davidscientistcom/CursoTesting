Imagenes y Dockerfiles

Una imagen no es un contenedor ejecutandose. Es la definicion a partir de la cual podras crear contenedores identicos las veces que haga falta.

1. La idea basica: una imagen define, un contenedor ejecuta

Si haces un paralelismo demasiado simple, acabas confundiendo conceptos. Asi que vamos directos al mecanismo: una imagen es un artefacto inmutable construido por capas; un contenedor es una instancia de ejecucion creada a partir de esa imagen. Cuando cambias el Dockerfile y reconstruyes, no estas editando el contenedor existente. Estas generando una nueva imagen desde la que luego podras lanzar contenedores nuevos.

Imagen
  • Se construye con docker build o a traves de Compose.
  • Encadena capas producidas por instrucciones como FROM, RUN y COPY.
  • No tiene procesos vivos por si misma.
Contenedor
  • Se crea a partir de una imagen.
  • Tiene un sistema de archivos efimero propio, salvo donde uses volumenes o mounts.
  • Ejecuta un proceso principal definido por CMD o ENTRYPOINT.

2. Ejemplo base minimo para entender el patron

Antes de mirar Jenkins, conviene ver un Dockerfile pequeno donde cada linea sea facil de leer. Este ejemplo no es el laboratorio real; es el escalon previo para que el real no te caiga encima de golpe.

dockerfile - Ejemplo base 
FROM python:3.12-slim

WORKDIR /app
COPY app.py /app/app.py

CMD ["python", "/app/app.py"]
1
FROM fija la base

Si haces esto, tu imagen empieza heredando Python 3.12 slim. Si eliges otra base, todo lo demas cambia: paquetes disponibles, tamano y herramientas de sistema.

2
WORKDIR fija el directorio de trabajo

Si no lo defines, las rutas de COPY y del proceso principal pueden acabar siendo mas confusas de lo necesario.

3
COPY mete codigo en la imagen

Lo que copies aqui queda dentro de la imagen. Si luego cambias el archivo local, la imagen no cambia hasta que la vuelvas a construir.

4
CMD define el proceso principal por defecto

Cuando el contenedor arranca, Docker ejecuta ese comando. Si el proceso termina, el contenedor tambien termina.

3. Caso real 1: la imagen de Jenkins del laboratorio

El ejemplo base sirve para entender la sintaxis. El laboratorio real obliga a dar un paso mas: necesitamos una imagen de Jenkins que venga preparada con herramientas y configuracion inicial para no depender de clics manuales.

dockerfile - Jenkins del laboratorio 
FROM jenkins/jenkins:lts-jdk21

USER root

RUN apt-get update \
    && apt-get install -y --no-install-recommends curl git docker.io \
    && rm -rf /var/lib/apt/lists/*

COPY plugins.txt /usr/share/jenkins/ref/plugins.txt
RUN jenkins-plugin-cli --plugin-file /usr/share/jenkins/ref/plugins.txt

COPY ref/init.groovy.d/ /usr/share/jenkins/ref/init.groovy.d/

USER jenkins

Que hace cada bloque y por que existe

FROM jenkins/jenkins:lts-jdk21: no partimos de una imagen vacia. Aprovechamos la distribucion oficial de Jenkins para no reinventar la base.

USER root: es necesario porque vamos a instalar paquetes del sistema. Si intentaras ese apt-get install siendo el usuario jenkins, el build fallaria por permisos.

RUN apt-get ... docker.io: aqui aparece una decision importante del laboratorio. Jenkins no solo ejecuta codigo; tambien necesita poder hablar con Docker para levantar contenedores durante el pipeline.

COPY plugins.txt y jenkins-plugin-cli: convierten la seleccion de plugins en parte de la imagen. Si los instalases a mano desde la web, perderias repetibilidad.

COPY ref/init.groovy.d/: deja scripts de inicializacion dentro de la imagen para que el Jenkins del laboratorio arranque ya preparado.

USER jenkins: tras instalar lo necesario, se vuelve a un usuario menos privilegiado. No es un adorno; es una reduccion del alcance de permisos en tiempo de ejecucion.

WARN
Error frecuente: olvidar volver a USER jenkins. El contenedor seguiria funcionando, pero habrias dejado Jenkins ejecutandose como root sin necesidad.

4. Caso real 2: la imagen minima del relay

El segundo Dockerfile del laboratorio sirve para enseñar lo contrario: a veces no hace falta una imagen pesada ni un gran numero de instrucciones. Si el servicio es pequeno, la imagen puede ser pequena y directa.

dockerfile - Webhook relay 
FROM python:3.12-slim

WORKDIR /app
COPY relay.py /app/relay.py

CMD ["python", "/app/relay.py"]

Este Dockerfile muestra que no siempre necesitas instalar dependencias de sistema, crear usuarios o copiar arboles grandes. Si el servicio hace poco y lo hace bien, su imagen puede ser corta y legible. Para clase es importante mostrar los dos extremos: una imagen compleja porque el servicio lo exige, y una imagen simple porque el servicio no necesita mas.

5. Construccion real observada en la sesion

En vez de hablar en abstracto de docker build, el laboratorio se reconstruyo con Compose. Eso permite ver a la vez que imagenes se construyen y como se nombran dentro del proyecto.

bash - Reconstruir solo las imagenes del lab 
cd /home/deivit/Lucentia/projects/jenkins-lab
docker compose build jenkins webhook-relay

Durante la revalidacion se observaron estos nombres de imagen:

text - Imagenes generadas 
jenkins-lab-jenkins
jenkins-lab-webhook-relay

Si cambias una linea del Dockerfile de Jenkins y haces docker compose up -d sin --build, seguiras ejecutando la imagen antigua. Este detalle es fuente habitual de confusion en clase: el codigo del Dockerfile puede estar correcto y aun asi no verse reflejado porque no se reconstruyo la imagen.

6. Que significa que una imagen se construya por capas

Cada instruccion relevante del Dockerfile genera una capa reutilizable. Si una capa no cambia, Docker puede reutilizarla. En la salida observada del build aparecieron varias lineas CACHED para Jenkins. Eso no es ruido: es la evidencia de que Docker no recompilo lo que seguia siendo identico.

INFO
Si haces esto: cambias relay.py y reconstruyes el relay, Docker invalida las capas a partir del COPY relay.py. Si haces lo otro: cambias solo el comando de ejecucion, la base python:3.12-slim sigue pudiendo reutilizarse.

7. Fallos tipicos cuando empiezas con Dockerfiles

FAIL
Fallo 1: copiar ficheros que no estan en el contexto de build. Si el archivo no esta dentro del directorio que Docker usa como contexto, COPY fallara aunque la ruta te parezca correcta.
FAIL
Fallo 2: instalar paquetes como usuario sin privilegios. En el Dockerfile real de Jenkins eso obligo a subir a USER root antes del apt-get install.
WARN
Fallo 3: creer que editar el Dockerfile cambia automaticamente los contenedores ya creados. No cambia nada hasta que reconstruyes la imagen y vuelves a crear o reiniciar el contenedor de forma adecuada.

8. Cierre del capitulo

  • Una imagen es la definicion repetible; un contenedor es la ejecucion concreta.
  • El laboratorio combina un Dockerfile complejo y uno minimo para que veas dos patrones reales.
  • La eleccion de usuario y de paquetes del sistema tiene consecuencias directas en permisos y comportamiento.
  • El siguiente paso es entender como esas imagenes se conectan entre si y donde guardan o no guardan estado.
