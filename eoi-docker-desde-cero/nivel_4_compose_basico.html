Compose base: leer y ejecutar el YAML sin magia

Compose no añade magia sobre Docker. Lo que hace es declarar en un solo fichero la topologia que de otro modo tendrias que reconstruir a mano comando a comando.

1. Que problema resuelve Compose

Si un laboratorio tuviera un solo contenedor, podrias vivir bastante tiempo con docker run. En cuanto aparecen varios servicios, puertos, volumenes, redes y variables de entorno, repetir todo a mano se vuelve fragil. Compose convierte esa secuencia manual en una declaracion versionable.

Sin Compose
  • Tienes que recordar orden de arranque, flags y nombres.
  • Un olvido pequeno cambia el comportamiento total del sistema.
  • Explicar el laboratorio en clase exige reconstruir demasiados detalles operativos.
Con Compose
  • El modelo entero queda descrito en YAML.
  • Puedes revisarlo, versionarlo y discutirlo antes de arrancar nada.
  • Los comandos de ciclo de vida se simplifican a up, down, ps y logs.

2. Ejemplo base: un compose pequeno para fijar la sintaxis

Antes del laboratorio real, conviene ver el esqueleto minimo de un fichero Compose. Asi puedes identificar con claridad que es una seccion raiz y que es una configuracion especifica de servicio.

yaml - Compose minimo 
services:
  web:
    image: nginx:1.27
    ports:
      - "8080:80"

volumes: {}
networks: {}

Ese ejemplo sirve para fijar el patron, pero todavia no enseña cooperacion real entre servicios. Para eso volvemos al laboratorio validado.

3. Anatomia del Compose real del laboratorio

El archivo docker-compose.yml de Jenkins Lab contiene todos los ingredientes que interesan en una clase: una imagen remota, dos builds locales, puertos, variables, volumenes, red propia y un healthcheck.

Campo Que expresa Ejemplo real
image Usar una imagen ya existente gitea/gitea:1.22.6
build Construir una imagen localmente context: ./jenkins
ports Publicar puertos hacia el host 9090:8080
environment Inyectar variables al proceso JENKINS_URL=http://jenkins:8080
volumes Persistencia o acceso a rutas del host jenkins_data:/var/jenkins_home
depends_on Orden de arranque basico relay depende de Jenkins
healthcheck Criterio observable de salud curl -fsS http://localhost:8080/login

4. El trozo de YAML que conviene leer despacio

yaml - Recorte representativo del laboratorio 
services:
  gitea:
    image: gitea/gitea:1.22.6
    container_name: gitea-lab
    ports:
      - "3000:3000"
      - "222:22"
    volumes:
      - gitea_data:/data

  jenkins:
    build:
      context: ./jenkins
    container_name: jenkins-lab
    ports:
      - "9090:8080"
      - "50000:50000"
    depends_on:
      - gitea

  webhook-relay:
    build:
      context: ./webhook-relay
    depends_on:
      - jenkins

Lee este fragmento siguiendo una regla simple: cada servicio responde a tres preguntas. De que imagen sale, que necesita para funcionar y como se expone o conecta con el resto. Si una de esas tres cosas no esta clara al explicarlo, la lectura del YAML todavia es superficial.

5. Comandos de ciclo de vida que si debes enseñar

Para una clase en directo conviene limitarse a los comandos que realmente cambian el estado del sistema y cuyo efecto pueda comprobarse enseguida.

1
docker compose config

Sirve para validar y expandir la configuracion final. Si el YAML tiene un problema estructural, aqui conviene detectarlo antes del arranque.

2
docker compose up -d --build

Construye lo que haga falta y deja el stack corriendo. Si haces cambios en Dockerfiles y no usas --build, puedes estar ejecutando una imagen vieja.

3
docker compose ps

Muestra si los servicios existen realmente y en que estado estan. No sustituye a los logs, pero es el primer filtro de validacion.

4
docker compose down -v --remove-orphans

Desmonta el laboratorio y, al usar -v, borra tambien los volumenes del proyecto. Si lo ejecutas, estas pidiendo una vuelta al estado cero.

6. Evidencia observable que ya tienes antes de abrir Jenkins

En esta sesion, la propia shell ya dejo varias pruebas de que Compose habia materializado el modelo: imagenes construidas, volumenes creados, red creada y servicios arrancados. Es importante decir esto en clase porque muchos alumnos tienden a pensar que Docker solo empieza a ser real cuando abren una interfaz web.

7. Dos comparaciones que aclaran mucho

up -d sin --build
  • Mas rapido si no has tocado imagenes.
  • Peligroso si acabas de editar un Dockerfile.
  • Puede llevarte a pensar que tu cambio no funciona cuando en realidad no llego a construirse.
up -d --build
  • Revisa las imagenes locales antes de arrancar.
  • Es la opcion segura para demostraciones donde acabas de cambiar algo del entorno.
  • Fue la opcion usada en esta revalidacion.

8. Fallos tipicos al trabajar con Compose

FAIL
Fallo 1: cambiar un nombre de servicio y no actualizar las referencias internas. Si el relay espera http://jenkins:8080 y renombras el servicio, el DNS interno deja de coincidir.
FAIL
Fallo 2: usar down -v sin querer. Si lo ejecutas, destruyes tambien los volumenes. En un laboratorio puede interesarte; en un entorno persistente hay que pensarlo antes.
WARN
Fallo 3: mirar solo la interfaz web y no mirar la salida de Compose. Cuando algo no arranca, la linea util suele estar antes en el terminal que en el navegador.

9. Cierre del capitulo

  • Compose sirve para declarar y repetir topologias, no solo para ahorrar escritura.
  • Un YAML se entiende mejor si lo lees como una descripcion operativa del sistema.
  • Los comandos utiles en clase son pocos, pero cada uno tiene un efecto muy concreto que debe explicarse.
  • El ultimo paso del curso es ver como convertir todo esto en una demo repetible de principio a fin, con esperas activas y bootstrap automatizado.
