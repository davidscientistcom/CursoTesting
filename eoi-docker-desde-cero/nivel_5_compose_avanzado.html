Repetibilidad y clase en directo

La parte mas delicada de un laboratorio no es que una vez funcione. Es que puedas repetirlo delante de una clase y saber que paso si algo no sale como esperabas.

1. Compose por si solo no resuelve toda la orquestacion practica

Compose arranca servicios, pero no garantiza que una aplicacion este lista para recibir trafico solo porque su proceso haya comenzado. En el laboratorio, ese matiz importa mucho: Jenkins puede estar arrancado como proceso y aun no estar listo para autenticarse ni para servir el job. Por eso aparecen healthchecks, scripts de espera y bootstrap automatizado.

INFO
Idea clave: depends_on ordena arranques basicos, pero no sustituye una comprobacion real de disponibilidad. Para clase, esta diferencia es importante porque explica muchos falsos arranques "verdes" que luego fallan en el paso siguiente.

2. El healthcheck real de Jenkins

El Compose del laboratorio declara un healthcheck para Jenkins. Esa configuracion es la razon por la que en docker compose ps se llego a ver healthy en la sesion de revalidacion.

yaml - Healthcheck real de Jenkins 
healthcheck:
  test: ["CMD", "curl", "-fsS", "http://localhost:8080/login"]
  interval: 20s
  timeout: 10s
  retries: 6
  start_period: 60s

Si haces la comprobacion contra /login y responde correctamente, tienes una evidencia bastante razonable de que Jenkins ya es util para el siguiente paso. Si la comprobacion falla, Compose no te dice solo "el proceso sigue vivo"; te dice que el servicio aun no esta sano.

3. Espera activa: no seguir hasta que Jenkins responda

Aun con healthcheck, el flujo docente necesita un punto de espera explicito antes del bootstrap. El script real hace precisamente eso: consulta Jenkins varias veces hasta que consigue autenticar contra /login.

bash - wait-for-jenkins.sh 
for attempt in $(seq 1 60); do
    if curl -fsS -u "$auth" "$jenkins_url/login" >/dev/null 2>&1; then
        echo "Jenkins responde en $jenkins_url"
        exit 0
    fi
    sleep 5
done

echo "Jenkins no estuvo listo a tiempo" >&2
exit 1

El mensaje Jenkins responde en http://localhost:9090 aparecio en la revalidacion de esta sesion. Esa linea es una evidencia importante porque marca el momento exacto a partir del cual tiene sentido seguir con el bootstrap del laboratorio.

4. Bootstrap automatico de Gitea

Una vez Jenkins y Gitea responden, el laboratorio crea automaticamente el usuario, el repositorio y el webhook. Esa automatizacion evita clics manuales, reduce errores y permite que la clase se concentre en las ideas de Docker en vez de quedarse atrapada en formularios web.

bash - Pasos clave de bootstrap-gitea.sh 
wait_for_gitea
ensure_user "$repo_owner" "$repo_password" "${repo_owner}@jenkins-lab.local"
ensure_repo
ensure_webhook

Durante la revalidacion quedaron confirmadas estas salidas:

text - Bootstrap observado 
Gitea responde en http://localhost:3000
Repositorio alumno/jenkins-python-lab creado
Webhook creado hacia http://webhook-relay:8081/gitea/push

5. El guion integral que conviene enseñar

El laboratorio ya empaqueta toda la secuencia en un solo script. Para clase, esto es util por dos razones: reduce cambios de contexto y deja por escrito el orden correcto de ejecucion.

bash - run-end-to-end-check.sh 
docker compose down -v --remove-orphans
docker compose up -d --build

"$LAB_ROOT/scripts/wait-for-jenkins.sh"
"$LAB_ROOT/scripts/bootstrap-gitea.sh"
"$LAB_ROOT/scripts/run-demo.sh"

Leido despacio, este script resume medio curso. Primero destruye el estado previo. Luego reconstruye el stack. Despues espera a Jenkins. A continuacion prepara Gitea. Finalmente ejecuta una demo que prueba que el laboratorio no solo arranca: tambien sirve para una historia completa de desarrollo y validacion.

6. La demo final: fallo real y reparacion real

El script run-demo.sh prepara un repositorio temporal, introduce un bug SQL, hace push, espera al build fallido, repara el bug y hace un segundo push. Lo importante no es Jenkins en si; lo importante para este curso es que Docker sostiene todo el circuito que permite observar ese comportamiento.

bash - Salida relevante de la revalidacion 
Build con fallo verificado: 1
Build reparado y verificado: 2
FAIL_BUILD=1
SUCCESS_BUILD=2

En una validacion anterior del propio laboratorio quedaron registrados tambien los builds 3 y 4. Eso confirma una idea importante para clase: no dependes de una unica ejecucion afortunada. El comportamiento se ha repetido en distintas reconstrucciones del entorno.

7. Secuencia docente recomendada para una clase en directo

1
Mostrar el compose antes de ejecutarlo

Asi el alumno sabe que deberia aparecer: servicios, puertos, volumenes y red.

2
Ejecutar run-end-to-end-check.sh

Permite recorrer el ciclo entero sin cambiar continuamente de comandos ni perder el hilo.

3
Parar en las evidencias visibles

Red creada, volumenes creados, Jenkins healthy, Gitea accesible, repo creado, webhook instalado.

4
Explicar el fallo del primer build

Es la mejor manera de demostrar que el entorno no esta maquillado para salir siempre verde.

5
Cerrar con la reparacion

El segundo build muestra que el laboratorio sigue vivo, permite iterar y refleja cambios reales de codigo.

8. Tres limites reales que conviene declarar

WARN
Limite 1: el laboratorio hace concesiones docentes, como desactivar CSRF en Jenkins y permitir determinados hosts internos en Gitea. Son decisiones validas para un entorno aislado, no para produccion.
WARN
Limite 2: al usar down -v, los contadores de builds pueden reiniciarse. En esta sesion quedaron en 1 y 2; en la validacion previa quedaron en 3 y 4. Lo importante es el patron de comportamiento, no el numero absoluto.
FAIL
Limite 3: si el host no tiene Docker operativo o si los puertos de Gitea y Jenkins estan ocupados, el laboratorio no puede arrancar aunque el material este bien escrito.

9. Cierre del curso

  • Docker no se ha presentado como una lista de comandos, sino como una forma de declarar y reconstruir un entorno completo.
  • Has visto ejemplos base y, despues, el caso real del laboratorio validado.
  • La repetibilidad no sale gratis: requiere healthchecks, esperas activas y bootstrap automatizado.
  • El material queda preparado para usarse en clase con comandos, salidas y fallos explicados paso a paso.
